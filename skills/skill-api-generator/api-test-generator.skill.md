# ZA Bank 接口自动化测试生成器

## 技能描述

这是一个**通用的接口自动化测试生成器**,适用于ZA Bank测试框架中的所有微服务接口测试。

**核心特性**:
- ✅ **服务无关**: 支持所有微服务(zabank-imc-activity、zabank-imc-cubercore、zabank-act-core、zabank-bms-batch等)
- ✅ **格式统一**: 基于统一的接口文档格式自动生成测试代码
- ✅ **智能适配**: 自动识别服务类型并选择正确的service层和scenario层
- ✅ **完整生成**: 一次性生成测试代码、测试数据、执行脚本
- ✅ **多环境支持**: 支持sit/auto_qe/uat等多环境配置

## 使用场景

当用户需要:
1. **为任何微服务接口编写自动化测试**
2. **基于接口文档快速生成测试套件**
3. **生成标准化的测试数据文件**
4. **快速测试和验证某个接口**

## 输入格式

用户需要提供以下格式的接口文档(Markdown文件):

```markdown
## 接口X: 接口名称

### 接口信息
- **接口路径**: `/path/to/api`
- **请求方法**: POST/GET
- **功能描述**: 接口功能说明

### 请求参数
| 参数名 | 类型 | 必填 | 说明 | 示例值 |
|--------|------|------|------|--------|
| param1 | String | 是 | 参数说明 | example |

### 响应参数
| 字段名 | 类型 | 说明 | 示例值 |
|--------|------|------|--------|
| code | String | 响应码 | 000000 |

### 接口X测试用例

#### 功能测试
##### 1. 正常场景测试 - 场景描述
- **用例描述**: ...
- **前置条件**: ...
- **请求参数**: ```json {...} ```
- **预期结果**: ```json {...} ```

#### 异常测试
##### 1. 参数错误测试
...
```

## 技能实现

### 第一步: 分析输入并识别服务

当用户提供文档或接口信息时:

1. **读取并解析文档**
   - 提取所有接口信息(路径、方法、参数、响应)
   - 提取所有测试用例场景
   - 识别优先级(P0/P1/P2)

2. **识别目标服务**
   - 从接口路径自动识别服务类型
   - 查找对应的service目录
   - 查找对应的scenario类

**服务识别规则**:
```python
# 路径前缀 -> 服务映射
"/activity/*" -> zabank_imc_activity_service
"/rc/*" -> zabank_rcs_core (RCS权益服务)
"/cuber/*" -> zabank_imc_cubercore_service
"/reward/*" -> zabank_imc_reward_service
"/act/*" -> zabank_act_core_service
"/batch/*" -> zabank_bms_batch_service
```

### 第二步: 动态获取服务信息

基于识别的服务,自动获取:

1. **Service层信息**
```python
# 读取service目录下的__init__.py获取接口映射
service_path = f"service/{service_name}/__init__.py"
# 解析mapping字典,获取可用的接口方法和Model类

# ⚠️ 如果Service层没有对应的接口方法:
# 方案1: 自动创建Service层代码
# 方案2: 提示用户需要先添加Service层接口定义
```

2. **Scenario层信息**
```python
# 查找对应的scenario文件
scenario_files = {
    "zabank_imc_activity_service": "scenario/imc/activity_scenario.py",
    "zabank_imc_cubercore_service": "scenario/imc/cubercore_scenario.py",
    "zabank_imc_reward_service": "scenario/imc/reward_scenario.py",
    "zabank_imc_interestcore_service": "scenario/imc/interestcore_scenario.py",
}
```

3. **配置信息**
```python
# 从config文件获取服务host配置
config_path = "config/uat.yaml"  # 或其他环境
```

### 第三步: 检查并添加Scenario Fixture

**重要**: 必须在`conftest.py`中添加对应的scenario fixture，否则测试会失败！

```python
# conftest.py
from scenario.imc.activity_scenario import ActivityScenario

@pytest.fixture(scope="session")
def activity_sc(config):
    return ActivityScenario(config)
```

**Fixture命名规则**: `{service_domain}_sc`
- zabank_imc_activity_service → activity_sc
- zabank_imc_cubercore_service → cuber_core_sc
- zabank_imc_reward_service → reward_sc

### 第四步: 生成测试代码

**关键原则**:
- ✅ **每个测试方法使用独立的YAML文件** (避免参数化问题)
- ✅ **使用`data[0]`获取测试数据** (YAML的tests列表中的第一个case)
- ✅ **不在测试文件中定义fixture** (统一在conftest.py管理)

#### 模板结构

```python
"""
@Project : zabank_imc_case
@File    : test_{interface_name}.py
@Author  : Auto Generated by API Test Generator
@Date    : {date}
"""
import allure
import pytest
from pytest_zati_base.utils.assertion import Assertion
from pytest_zati_base.utils import logging

from service.{service_name} import model


@allure.epic("{epic_name}")
@allure.feature("{feature_name}")
class Test{InterfaceName}:
    """接口测试类"""

    @allure.title("{test_title}")
    @allure.story("{story_name}")
    @pytest.mark.data('{data_file_path}')  # 独立的YAML文件
    @pytest.mark.{priority}
    def test_{interface_name}_{scene_number}(self, {fixture_name}, data):
        """
        测试场景: {scene_description}

        测试步骤:
        {test_steps}

        预期结果:
        {expected_results}
        """
        # 步骤1: 获取测试数据(从tests列表的第一个case)
        step = data[0]

        # 步骤2: 构造请求
        user_id = step.get("user_id", "")
        header = {{"userid": user_id, "language": "ca"}}
        params = model.{ModelClassName}()
        params.json_data = {{
            {request_params}
        }}

        # 步骤3: 发送请求
        logging.info(f"测试{scene_description}")
        resp = {fixture_name}.controller.{controller_method}(load=params, headers=header)

        # 步骤4: 验证响应码(只检查code字段)
        expected_code = step.get("expected_code", "000000")
        Assertion.str_assert(resp.json()["code"], expected_code)
        logging.info(f"响应码验证通过: {{resp.json()['code']}}")
```

```python
# 根据服务自动填充:

```

### 第五步: 生成测试数据YAML

**核心原则**:
- ✅ **1个测试方法 = 1个YAML文件 = 1个case** (避免参数化导致重复运行)
- ✅ **文件命名**: `{interface_name}_{scene_number}_{scene_name}.yaml`
- ✅ **必须使用tests列表格式** (框架要求)

#### YAML文件模板

```yaml
# data/{service_module}/{interface_name}_{scene_number}_{scene_name}.yaml
# 示例: data/za_zone/entry_page_01_joined.yaml

# ===== 环境配置(可选,正常/边界场景需要) =====
env:
  sit:
    user_id: "{sit_user_id}"
    customer_no: "{sit_customer_no}"

  auto_qe:
    user_id: "{autoqe_user_id}"
    customer_no: "{autoqe_customer_no}"

  uat:
    user_id: "{uat_user_id}"
    customer_no: "{uat_customer_no}"

# ===== 测试用例(必须,且只包含1个case) =====
tests:
  - case: {场景描述}
    data:
      - step: {步骤名称}
        input:
          user_id: ${user_id}         # 使用${}引用环境变量
          customer_no: ${customer_no}
          {param1}: "{value1}"
        expected:
          expected_code: "{期望响应码}"
```

#### 示例1: 正常场景YAML (需要env配置)


```yaml
# data/za_zone/entry_page_04_invalid.yaml

tests:
  - case: 客户不存在
    data:
      - step: 客户不存在

          user_id: "9999999999999999"

```

**文件组织**:
```
data/za_zone/
├── entry_page_01_joined.yaml       # 测试方法1的数据
├── entry_page_02_not_joined.yaml   # 测试方法2的数据
├── entry_page_03_relative.yaml     # 测试方法3的数据

```

**关键点**:
- ✅ 每个YAML只包含1个case,避免pytest参数化
- ✅ tests必须是列表格式,即使只有1个case
- ✅ 测试代码使用`data[0]`获取唯一的case
- ✅ 正常/边界场景需要env配置多环境数据
- ✅ 异常场景使用固定无效值,不需要env配置

## 智能特性

### 1. 接口方法名自动推断

根据接口路径自动生成controller方法名:

```python
# 规则:
# /dmb/nok9iy/activity/zazone/getVerifyCode

def path_to_method_name(api_path):
    # 提取路径中的关键部分
    # 转换为snake_case
    # 匹配service的__init__.py中的mapping
    return method_name
```

### 2. Model类名自动推断

```python
# 规则:
# activity_za_zone_get_verify_code
# -> ActivityZaZoneGetVerifyCode

def method_to_model_name(method_name):
    # 转换为PascalCase
    return model_class_name
```

### 3. 测试数据智能生成

根据参数类型和场景自动生成**多环境**测试数据:

```python
def generate_test_data(param_info, scenario_type):
    """
    生成多环境YAML格式的测试数据

    Args:
        param_info: 参数信息(类型、必填性等)
        scenario_type: 场景类型(normal/boundary/exception)

    Returns:
        dict: 包含env配置和tests用例的YAML数据结构
    """

    # 环境配置部分
    env_config = {
        "sit": {},
        "auto_qe": {},
        "uat": {}
    }

    # 根据场景类型生成用户变量
    if scenario_type in ["normal", "boundary"]:
        # 正常和边界场景使用环境变量
        for env in ["sit", "auto_qe", "uat"]:
            env_config[env] = {
                "user_id": f"${{{scenario_type}_user_id}}",
                "customer_no": f"${{{scenario_type}_customer_no}}"
            }

    # 特殊字段处理
    special_fields = {
        "email": "\"test@example.com\"",
        "mobile": "\"12345678\"",
        "userId": "${user_id}",  # 引用环境变量
        "customerNo": "${customer_no}",  # 引用环境变量
    }

    # 异常场景使用固定无效值
    invalid_values = {
        "email": "\"invalid-email-format\"",
        "customerNo": "\"INVALID_CUSTOMER_9999999\"",
        "userId": "\"999999999999999\"",
    }
```

### 4. 断言自动生成

**重要**: 结果验证只检查`code`字段,保持简洁高效:

```python
# 响应码断言(唯一必需的断言)
expected_code = step.get("expected_code", "000000")
Assertion.str_assert(resp.json()["code"], expected_code)

# 可选: 记录响应内容供调试
logging.info(f"响应码: {resp.json()['code']}")
logging.info(f"响应数据: {resp.json()}")
```

**设计原则**:
- ✅ 只验证`code`字段,简化测试逻辑
- ✅ 使用`.get()`方法支持环境变量替换
- ✅ 添加日志记录便于问题定位
- ❌ 不验证业务字段(如tokenId、currentLevel等)
- ❌ 不进行数据库断言(除非特殊需求)

## 多环境测试数据规范⭐

### YAML文件结构

**必须采用以下标准格式**:

```yaml
# ===== 环境配置部分 =====
env:
  sit:
    # 场景相关的测试账号
    {scenario}_user_id: "{实际user_id}"
    {scenario}_customer_no: "{实际customer_no}"

  auto_qe:
    {scenario}_user_id: "{实际user_id}"
    {scenario}_customer_no: "{实际customer_no}"

  uat:
    {scenario}_user_id: "{实际user_id}"
    {scenario}_customer_no: "{实际customer_no}"

# ===== 测试用例部分 =====
tests:
  - case: {用例名称}
    data:
      - step: {步骤名称}
        input:
          user_id: ${变量名}  # 引用env中的变量
          customer_no: ${变量名}
          param1: "value1"
        expected:
          expected_code: "000000"
        description: "{场景说明}"
```

### 变量引用规则

1. **环境变量引用**: `${变量名}` - 测试框架会根据`--envId`自动替换
2. **变量命名**: 使用`{场景}_user_id`格式,便于区分不同场景的测试账号
3. **固定值**: 异常场景直接写固定的无效值,不使用环境变量

### 测试代码适配

测试代码中使用`.get()`方法获取数据,支持环境变量替换:


## 使用最佳实践

1. **文档先行**: 确保接口文档完整准确
2. **分步骤生成**: 先生成一个接口测试,验证通过后再批量生成
3. **及时review**: 生成后检查代码,进行必要调整
4. **持续维护**: 接口变更时同步更新测试代码
5. **多环境数据管理**⭐:
   - 每个环境(sit/auto_qe/uat)配置独立的测试账号
   - 使用`${变量名}`引用环境配置
   - 正常/边界场景使用环境变量,异常场景使用固定无效值
   - 定期更新各环境的测试数据
6. **环境切换**:
   - 通过`--envId`参数切换测试环境
   - 在pytest.ini中配置默认环境
   - CI/CD中使用参数化环境配置
