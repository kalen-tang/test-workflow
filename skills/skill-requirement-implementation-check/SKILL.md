# 需求实现检查技能文档

## 需求实现检查与对齐分析报告生成

### 一、技能目标

本技能专门用于需求落地全流程的校验工作，适配需求文档、设计文档、代码差异文档分目录存放的场景，通过检查各目录下文档的完整性、规范性，对比三者间的需求实现对齐情况，最终生成标准化Word报告，提供文档质量评分、对齐颗粒度分析、针对性测试建议及风险解决方案，帮助团队把控需求落地质量，降低测试与交付风险。

### 二、主要功能

#### 1. 分目录多类型文档检查（需求/设计/代码差异文档）

当用户将需求文档、设计文档（含概要设计、详细设计）、代码差异文档分别存放在不同目录时，我将先识别各目录对应文档类型，再全面检查各文档质量，生成独立质量评分（满分100分）；支持单个目录下多份同类型文档（如设计目录下含概要设计、详细设计多版本文档）的批量检查，具体检查维度如下：

##### 1.1 需求文档检查（评分维度）

- 完整性（30分）：检查需求描述、业务场景、功能边界、输入输出、异常场景是否完整，无遗漏

- 明确性（25分）：检查需求表述是否清晰、无歧义，术语统一，无模糊性描述（如“大概”“可能”）

- 可行性（20分）：检查需求是否符合技术实现逻辑、项目资源及业务实际，无不可落地的需求

- 规范性（15分）：检查需求文档格式、编号、版本管理是否规范，符合团队文档标准

- 可测试性（10分）：检查需求是否可量化、可验证，每个需求点均有明确的验证标准

##### 1.2 设计文档检查（评分维度）

- 完整性（30分）：检查技术架构、模块划分、接口设计、数据库设计、异常处理设计是否完整

- 合理性（25分）：检查设计方案是否匹配需求文档，技术选型、模块交互、数据流向是否合理

- 详细度（20分）：检查设计细节是否足够（如接口参数、逻辑流程、异常分支），可直接指导开发

- 规范性（15分）：检查设计文档格式、图表规范、术语统一，符合技术文档标准

- 可维护性（10分）：检查设计方案是否具备良好的扩展性、可维护性，降低后续迭代成本

##### 1.3 代码差异文档检查（评分维度）

- 完整性（30分）：检查代码变更范围是否覆盖需求及设计要求，无遗漏的需求实现代码

- 规范性（25分）：检查代码格式、命名规范、注释完整性，符合团队编码规范

- 正确性（25分）：检查代码逻辑是否符合设计文档，无逻辑错误、语法错误，异常处理到位

- 安全性（20分）：检查代码是否存在安全隐患（如参数泄露、越权风险、漏洞等）

#### 2. 需求实现对齐对比分析

以需求目录下的需求文档为基准，逐层对比设计目录下的设计文档、代码差异目录下的代码差异文档，分析需求实现的对齐颗粒度，明确未对齐点及原因；支持多版本同类型文档的交叉对比（如需求V2.0与设计V2.0对齐、设计V2.0与代码差异V2.0对齐），具体对比维度如下：

##### 2.1 需求文档 vs 设计文档 对齐对比

- 每个需求点是否有对应的设计方案支撑，无需求漏设计情况

- 设计方案是否完全匹配需求描述，无设计偏离、设计过度或设计不足

- 需求中的业务规则、异常场景，是否在设计文档中明确对应的技术实现方案

- 对齐颗粒度分级（精准对齐/基本对齐/部分对齐/未对齐），标注未对齐明细及原因（明确对应文档所在目录及版本）

##### 2.2 设计文档 vs 代码差异文档 对齐对比

- 设计文档中的模块划分、接口设计、逻辑流程，是否在代码中准确实现

- 代码变更内容是否完全覆盖设计要求，无设计漏实现、代码与设计不一致情况

- 代码中的异常处理、数据交互，是否符合设计文档规范

- 对齐颗粒度分级（精准对齐/基本对齐/部分对齐/未对齐），标注未对齐明细及原因（明确对应文档所在目录及版本）

##### 2.3 整体对齐颗粒度汇总

综合两层对比结果，给出整体需求实现对齐颗粒度评分（满分100分），明确核心未对齐点、影响范围及整改优先级，区分“需求问题”“设计问题”“代码问题”三类原因，并标注各问题对应文档的所在目录及版本，便于快速定位整改。

#### 3. 测试建议生成

结合分目录文档质量检查结果、对齐对比分析，生成针对性、可落地的测试建议，覆盖测试重点、测试方法，适配不同测试阶段；建议中明确对应测试场景关联的文档目录及版本，提升测试针对性：

##### 3.1 测试重点建议

- 针对文档质量薄弱点（如需求不明确、设计不合理），明确测试重点关注场景，标注对应文档所在目录及版本

- 针对未对齐点、高风险点，优先规划测试用例，重点验证需求落地一致性，关联对应文档目录及版本

- 针对代码差异文档中的变更点，明确回归测试范围，避免遗漏关联功能，标注代码差异文档所在目录

##### 3.2 测试方法建议

- 功能测试：明确各未对齐点、核心需求的测试场景、前置条件、预期结果，关联对应文档目录及版本

- 回归测试：针对代码变更，建议回归范围及回归优先级，提升回归效率，标注代码差异文档所在目录

- 风险测试：针对文档中发现的漏洞、不合理设计，建议专项风险测试方案，标注对应文档所在目录及版本

- 自动化测试：推荐可自动化的测试场景，降低重复测试成本（如接口自动化、核心流程自动化）

#### 4. 风险点及高效解决方案

基于分目录文档检查、对齐分析结果，识别需求实现全流程中的测试风险、交付风险，分级标注风险等级（高/中/低），并给出高效、可落地的解决措施；风险点中明确对应文档所在目录及版本，便于快速定位整改：

##### 4.1 风险点识别（核心类别）

- 需求风险：需求不明确、不可行、存在歧义，导致测试用例设计困难、交付偏差（标注需求文档所在目录及版本）

- 设计风险：设计方案不合理、不完整，导致代码实现异常、测试返工（标注设计文档所在目录及版本）

- 代码风险：代码与设计/需求不一致、存在安全隐患、逻辑错误，导致测试不通过（标注代码差异文档所在目录）

- 测试风险：测试重点不明确、测试覆盖不足，导致线上问题遗漏（关联对应文档目录及版本）

##### 4.2 解决方案要求

- 针对性：每个风险点对应专属解决方案，不泛泛而谈，明确具体操作步骤，关联对应文档目录及版本

- 高效性：解决方案简洁、易落地，可快速解决问题，降低整改成本

- 可操作性：明确解决方案的执行主体、执行步骤、完成标准，便于团队落地

- 预防性：针对高风险点，额外给出预防措施，避免后续同类问题重复出现

#### 5. 标准化Word报告生成

整合上述所有分析结果，生成标准化Word报告，报告中明确标注各文档的所在目录及版本，结构清晰、内容完整，可直接用于团队评审、汇报，支持后续追溯及整改跟进，确保报告的专业性和实用性。

### 三、工作流程（适配分目录文档）

#### 步骤1：分目录文档接收与预处理

接收分目录文档（明确需求目录、设计目录、代码差异目录） → 识别各目录下文档类型、版本 → 校验各目录文档完整性（是否缺失必要文档、版本是否匹配） → 逐目录、逐文档进行质量检查 → 生成各文档独立质量评分及目录级质量汇总

#### 步骤2：需求实现对齐对比分析

以需求目录下的需求文档为基准 → 需求文档（对应目录+版本）vs 设计文档（对应目录+版本）对比 → 设计文档（对应目录+版本）vs 代码差异文档（对应目录+版本）对比 → 标注未对齐点、原因及对应文档目录+版本 → 分级对齐颗粒度 → 整体对齐评分

#### 步骤3：风险识别与方案制定

基于分目录文档检查+对齐分析 → 识别各类风险点（标注对应文档目录+版本） → 风险等级划分（高/中/低） → 制定针对性解决方案及预防措施 → 生成测试建议（关联对应文档目录+版本）

#### 步骤4：Word报告生成与输出

整合所有分析结果（明确标注各文档目录+版本） → 按标准化模板组织内容 → 生成Word报告（含评分、对齐分析、建议、方案） → 输出最终报告

### 五、使用指南（适配分目录文档）

1. 上传分目录文档: 明确提供需求目录、设计目录、代码差异目录的路径及对应文档（支持常见格式：doc/docx、txt、md等），标注各目录下文档的版本号及对应需求范围；若单个目录下有多个同类型文档，需明确各文档的用途及版本关联关系。

2. 指定检查重点: 可以指定关注的特定需求模块、设计模块或代码变更范围，明确是否需要重点检查某类目录下的文档（如仅检查需求目录与设计目录的对齐情况），或重点检查某目录下的特定版本文档。

3. 选择报告详细度: 可以要求详细分析（含完整扣分明细、未对齐点明细、风险点明细，标注对应文档目录及版本）或概要分析（仅保留核心结果、重点建议及方案）。

4. 定制化需求: 可以根据项目特点，提出特殊要求（如调整文档评分维度及权重、自定义Word报告模板、新增特定风险检查维度、适配特定目录命名规范等）。

5. 报告获取: 技能执行完成后，将输出标准化Word报告（含分目录文档明细），可直接下载、编辑、用于团队评审或汇报。

### 六、最佳实践建议（适配分目录文档）

#### 1. 分目录文档质量要求

- 需求目录：建议按团队标准模板编写文档，统一命名规范（如“需求-V2.0-XX模块.docx”），明确需求编号、版本、业务场景、验证标准，避免模糊性、歧义性描述；目录下仅存放需求相关文档，避免混杂其他类型文档。

- 设计目录：建议按“概要设计”“详细设计”分类存放文档，统一命名规范（如“设计-概要-V2.0-XX模块.docx”），包含完整的技术架构图、模块交互图、接口设计文档、数据库表设计，细节足够支撑开发实现；明确各设计文档与需求文档的版本对应关系。

- 代码差异目录：建议按需求模块或开发迭代版本分类存放文档，统一命名规范（如“代码差异-V2.0-XX需求.docx”），明确代码变更范围、对应需求编号、变更原因，标注核心变更点及影响范围，注释完整；目录下仅存放代码差异相关文档，便于快速定位。

#### 2. 需求实现检查策略

- 早期介入: 在设计文档完成后、代码开发初期，即可进行需求目录与设计目录的对齐检查，提前发现设计偏离需求问题，降低整改成本。

- 持续跟进: 代码开发过程中，可定期提交代码差异目录下的文档，进行阶段性对齐检查（设计目录 vs 代码差异目录），避免后期大规模未对齐。

- 风险驱动: 重点关注高风险未对齐点、高评分扣分项，明确对应文档所在目录及版本，优先整改，避免影响测试进度及交付质量。

- 闭环管理: 针对报告中的整改建议、风险解决方案，明确责任人及完成时限，整改完成后，按原目录结构提交修订后的文档，重新进行检查，形成闭环。

#### 3. 协作建议

- 与产品、开发团队明确分目录命名及文档管理规范，统一各目录下文档的版本管理规则，避免因目录混乱、版本不匹配导致检查偏差。

- 定期组织文档评审会，结合本技能生成的报告，同步分目录文档质量、对齐情况及风险点，协同推进整改；明确各目录文档的维护责任人。

- 建立文档质量基线、对齐检查标准及分目录管理规范，后续项目可复用，提升需求实现检查的效率和一致性。

- 测试团队可提前介入，结合报告中的测试建议，关联对应目录下的文档，提前规划测试用例，提升测试效率，覆盖核心风险点。
