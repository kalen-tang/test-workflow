# 测试左移接口分析与测试用例推荐

## 技能目标

本技能专门用于测试左移的前置工作，通过分析开发方案文档，提取关键接口信息，识别接口间的业务关联，生成单接口测试用例和场景测试用例，帮助测试团队在开发早期阶段介入，全面覆盖接口功能和业务流程，提升测试效率和质量。

## 主要功能
### 1. 提取网页信息
- 若命令中需要从网页中获取页面内容，则使用playwright从命令提供的链接中提取网页内容，包括标题、正文、图片等信息。
- 如果当前用户没有安装playwright的mcp，先进行安装：claude mcp add playwright npx @playwright/mcp@latest
- 如果用户提供了网页链接，但需要登录，则提示用户提供用户名密码

### 2. 开发方案文档分析
当用户提供开发方案文档时，我将：
- 全面解析文档结构和内容
- 识别文档中的技术架构信息
- 提取业务需求和功能描述
- 分析系统设计方案
- 检查文档质量
  - 开发方案文档应包含完整的接口设计信息
    - 文档中直接列出了接口定义
    - 或者文档中附上了接口链接，域名为udoc.in.za
  - 建议使用标准的API文档格式（如OpenAPI/Swagger）
  - 接口参数说明应详细完整
  - 包含接口错误码信息

### 3. 接口信息提取
从开发方案文档中提取以下接口信息，并确保入参出参和接口对应的上：
如果开发文档直接列出了接口定义，则直接提取
如果开发文档中只附上了接口的链接，域名为udoc.in.za，则需要到链接中提取接口定义，使用playwrite从链接中提取网页内容，该网站的账号：admin ，密码：Za123456 

#### 基础接口信息
- **接口名称**: API 接口的业务名称
- **接口路径**: RESTful API 的 URL 路径(需包含微服务域名+接口路径)
- **请求方法**: HTTP 方法（GET、POST、PUT、DELETE等）
- **功能描述**: 接口实现的业务功能

#### 接口路径校验规则
提取接口路径后,必须进行以下校验:

1. **网关接口检测**:
   - 如果接口路径中包含 `dmb`,则判定为网关接口
   - **必须提示用户**: "检测到网关接口,测试左移需要使用微服务接口,请替换为对应的微服务接口路径"
   - 停止该接口的后续分析,等待用户提供微服务接口

2. **微服务识别**:
   - 如果路径中不包含 `dmb`,则根据路径前缀自动识别所属微服务
   - 微服务映射规则如下:

   | 路径前缀 | 微服务名称 | 服务说明 |
   |---------|-----------|---------|
   | /activity/* | zabank_imc_activity_service | 活动服务 |
   | /rc/* | zabank_rcs_core | RCS权益服务 |
   | /cuber/* | zabank_imc_cubercore_service | Cuber核心服务 |
   | /reward/* | zabank_imc_reward_service | 奖励服务 |
   | /act/* | zabank_act_core_service | ACT核心服务 |
   | /batch/* | zabank_bms_batch_service | 批处理服务 |

3. **完整路径格式**:
   - 标准格式: `微服务域名 + 接口路径`
   - 示例: `zabank_imc_activity_service/activity/list`
   - 如果文档中只提供了接口路径,需要根据上述映射规则补充微服务域名

#### 详细参数信息
- **请求参数**:
  - 路径参数（Path Parameters）
  - 查询参数（Query Parameters）
  - 请求头参数（Headers）
  - 请求体参数（Request Body）
  - 参数类型、是否必填、默认值、取值范围

- **响应参数**:
  - 响应状态码
  - 响应头信息
  - 响应体结构
  - 数据类型和字段说明
  - 错误响应格式

### 4. 测试左移用例推荐

#### 单接口测试用例
- **正常流程测试**: 验证接口正常业务逻辑
- **边界值测试**: 参数边界值、极限值测试
- **异常流程测试**: 错误参数、异常场景测试
- **业务规则验证**: 业务逻辑正确性验证

#### 接口契约测试用例
- **请求参数校验**: 必填参数、参数格式、参数类型验证
- **响应格式验证**: 响应结构、字段类型、数据格式验证
- **状态码验证**: 各种场景下的HTTP状态码正确性
- **错误处理验证**: 异常情况下的错误响应格式

#### 场景测试用例（新增）
基于提取的多个接口，分析业务流程和数据依赖关系，设计跨接口的场景测试用例：

- **业务流程场景**:
  - 识别完整的业务流程（如：创建 → 查询 → 更新 → 删除）
  - 设计端到端的业务场景测试
  - 验证多个接口协作完成业务目标

- **数据依赖场景**:
  - 识别接口间的数据传递关系
  - 测试上游接口输出作为下游接口输入
  - 验证数据在流程中的完整性和一致性

- **状态流转场景**:
  - 识别业务状态机（如：订单状态、活动状态等）
  - 测试状态的正向流转和逆向流转
  - 验证非法状态流转的拦截

- **并发与时序场景**:
  - 多接口并发调用的一致性
  - 接口调用时序依赖测试
  - 幂等性验证

- **异常恢复场景**:
  - 中间步骤失败的回滚测试
  - 部分成功场景的数据一致性
  - 补偿机制验证

## 工作流程

### 步骤1：文档接收与预处理
```
接收开发方案文档 → 文档格式识别 → 内容结构化解析 → 文档内容分析 →  检查文档质量
```

### 步骤2：接口信息提取
```
接口定义识别 → 接口路径校验(检测dmb/识别微服务) → 参数信息提取 → 信息结构化整理 → 接口关系分析
```

### 步骤3：业务场景识别
```
分析业务流程 → 识别接口依赖关系 → 提取业务状态流转 → 识别数据传递链路 → 场景分类归纳
```

### 步骤4：测试用例设计
```
单接口用例设计 → 场景用例设计 → 接口风险评估 → 优先级排序
```

### 步骤5：输出与建议
```
生成完整的测试左移文档（包含单接口用例 + 场景用例）
```

## 场景识别指南

在生成场景测试用例前，需要先分析文档中的接口关系，识别可能的业务场景：

### 场景识别维度

1. **接口命名规律识别**:
   - CRUD操作: create/add/save → get/query/list → update/modify/edit → delete/remove
   - 状态流转: submit → review/audit → approve/reject → publish/close
   - 业务流程: register → verify → activate → use → expire/renew

2. **数据依赖关系识别**:
   - 查找返回ID类字段的接口（如: userId, orderId, activityId等）
   - 查找需要ID作为入参的接口
   - 建立数据流向关系链

3. **业务领域聚合**:
   - 同一业务领域的接口（如：活动相关、订单相关、用户相关）
   - 识别领域内的完整业务闭环

4. **状态机识别**:
   - 识别status、state类字段
   - 分析状态的可能流转路径
   - 识别状态流转的触发接口

### 场景提取示例

假设文档中有以下接口：
- `/activity/create` - 创建活动
- `/activity/get` - 查询活动详情
- `/activity/audit` - 审核活动
- `/activity/publish` - 发布活动
- `/activity/close` - 关闭活动

则可识别出场景：
1. **完整生命周期场景**: create → audit → publish → close
2. **查询验证场景**: create → get (验证创建结果)
3. **异常流程场景**: create → audit(reject) → publish(应失败)

## 使用指南

1. **上传文档**: 直接提供开发方案文档内容或文档路径
2. **指定分析重点**: 可以指定关注的特定模块或接口
3. **选择输出详细度**: 可以要求详细分析或概要分析
4. **定制化需求**: 可以根据项目特点提出特殊要求
5. **场景优先**: 如果文档包含多个相关接口，优先生成场景测试用例

## 最佳实践建议

### 文档质量要求
- 开发方案文档应包含完整的接口设计信息，或者udoc链接
- 建议使用标准的API文档格式（如OpenAPI/Swagger）
- 接口参数说明应详细完整

### 测试左移策略
- **早期介入**: 在需求评审阶段就开始测试分析
- **持续跟进**: 随开发进度更新测试用例
- **风险驱动**: 重点关注高风险、核心业务接口
- **自动化优先**: 推荐可自动化的测试用例设计
- **场景优先**: 优先设计端到端的业务场景用例，覆盖完整业务流程
- **分层测试**: 单接口用例保证接口稳定性，场景用例保证业务正确性

### 协作建议
- 与开发团队保持密切沟通
- 定期更新接口变更信息
- 建立接口测试基线
- 持续优化测试策略

### 场景用例设计原则
- **真实性**: 场景应反映真实的业务使用情况
- **完整性**: 覆盖业务流程的各个环节
- **可验证性**: 每个步骤都有明确的验证点
- **可维护性**: 场景描述清晰，易于理解和维护
- **数据关联**: 明确标注接口间的数据传递关系（如：使用{{步骤1返回的ID}}）

## 注意事项

1. **场景识别要点**:
   - 优先识别业务主流程，再识别异常分支
   - 关注接口的数据依赖关系，不是所有接口都需要组合成场景
   - 场景数量不宜过多，聚焦核心业务流程（通常3-5个场景即可）

2. **用例设计要点**:
   - 单接口用例聚焦接口自身的功能和契约
   - 场景用例聚焦业务流程和数据一致性
   - 每个场景用例应包含正向流程 + 至少1个异常流程

3. **数据传递标记**:
   - 使用 `{{步骤N返回的字段名}}` 标记数据依赖
   - 便于测试执行时进行数据提取和传递

通过本技能的应用，可以显著提升测试左移的效率和质量，帮助团队在开发早期发现和预防潜在问题，降低项目风险。同时，通过场景用例的设计，能够更全面地验证业务流程的正确性和系统的可靠性。
