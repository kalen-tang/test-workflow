# 需求实现检查技能

## 技能目标

检查需求文档、设计文档、代码差异文档分目录存放的完整性和规范性，对比三者间的需求实现对齐情况，生成结构化 Markdown 报告（包含文档质量评分、对齐分析、测试建议、风险方案）。

## 输入

接收三类文档目录（至少提供需求目录）：

1. **需求文档目录**（必须）：包含需求文档（doc/docx/txt/md）
2. **设计文档目录**（可选）：包含概要设计、详细设计文档
3. **代码差异文档目录**（可选）：包含代码变更分析文档

## 执行流程

> **进度追踪**：开始执行前，使用 TaskCreate 为每个步骤创建任务，每个步骤开始时标记 `in_progress`，完成后标记 `completed`。

### 步骤1：文档接收与预处理

1. 识别各目录下文档类型和版本
2. 校验各目录文档完整性（是否缺失必要文档、版本是否匹配）
3. 逐目录、逐文档进行质量检查
4. 生成各文档独立质量评分及目录级质量汇总

### 步骤2：文档质量检查

#### 需求文档评分维度（总分100分）

| 维度 | 分值 | 检查内容 |
|------|------|---------|
| 完整性 | 30分 | 需求描述、业务场景、功能边界、输入输出、异常场景是否完整 |
| 明确性 | 25分 | 表述是否清晰无歧义，术语统一，无模糊描述 |
| 可行性 | 20分 | 是否符合技术实现逻辑和项目资源 |
| 规范性 | 15分 | 格式、编号、版本管理是否规范 |
| 可测试性 | 10分 | 是否可量化、可验证 |

#### 设计文档评分维度（总分100分）

| 维度 | 分值 | 检查内容 |
|------|------|---------|
| 完整性 | 30分 | 架构、模块、接口、数据库、异常处理设计是否完整 |
| 合理性 | 25分 | 是否匹配需求，技术选型和数据流向是否合理 |
| 详细度 | 20分 | 细节是否足够支撑开发 |
| 规范性 | 15分 | 格式、图表、术语统一 |
| 可维护性 | 10分 | 扩展性和可维护性 |

#### 代码差异文档评分维度（总分100分）

| 维度 | 分值 | 检查内容 |
|------|------|---------|
| 完整性 | 30分 | 变更范围是否覆盖需求和设计要求 |
| 规范性 | 25分 | 代码格式、命名、注释完整性 |
| 正确性 | 25分 | 逻辑是否符合设计，异常处理是否到位 |
| 安全性 | 20分 | 是否存在参数泄露、越权等安全隐患 |

### 步骤3：需求实现对齐对比

以需求文档为基准，逐层对比：

1. **需求 vs 设计**：每个需求点是否有设计方案支撑，是否存在设计偏离/过度/不足
2. **设计 vs 代码**：设计中的模块、接口、逻辑是否在代码中准确实现
3. **整体评分**：综合两层对比，给出对齐颗粒度评分（满分100分）

对齐等级：精准对齐 / 基本对齐 / 部分对齐 / 未对齐

### 步骤4：风险识别与测试建议

**风险分类**：
- 需求风险：不明确、不可行、存在歧义
- 设计风险：不合理、不完整
- 代码风险：与设计/需求不一致、安全隐患
- 测试风险：覆盖不足、重点不明确

**测试建议**：功能测试、回归测试、风险测试、自动化测试，按优先级排列。

### 步骤5：报告生成与输出

整合所有分析结果，按标准化模板输出 Markdown 报告。

## 输出

报告输出到 `./result/` 目录，文件名：`<需求ID>_需求实现检查报告.md`

> 报告模板参见 `references/report-template.md`

## 使用指南

1. **上传文档**：提供需求目录、设计目录、代码差异目录路径，标注各目录文档版本
2. **指定重点**（可选）：指定关注的需求模块或代码变更范围
3. **选择详细度**（可选）：详细分析（完整扣分明细）或概要分析（仅核心结果）
4. **定制化**（可选）：调整评分维度权重、自定义报告模板

## 最佳实践

- **早期介入**：设计文档完成后即可进行需求与设计的对齐检查
- **持续跟进**：开发过程中定期提交代码差异文档进行阶段性对齐检查
- **风险驱动**：优先整改高风险未对齐点
- **闭环管理**：整改完成后重新检查形成闭环
