# 场景测试用例识别指南

## 场景测试用例类型

基于提取的多个接口，分析业务流程和数据依赖关系，设计跨接口的场景测试用例。

### 1. 业务流程场景

**目标**：
- 识别完整的业务流程（如：创建 → 查询 → 更新 → 删除）
- 设计端到端的业务场景测试
- 验证多个接口协作完成业务目标

**示例**：
- 活动管理流程：创建活动 → 审核活动 → 发布活动 → 查询活动 → 关闭活动
- 订单流程：创建订单 → 支付订单 → 发货 → 确认收货 → 评价

### 2. 数据依赖场景

**目标**：
- 识别接口间的数据传递关系
- 测试上游接口输出作为下游接口输入
- 验证数据在流程中的完整性和一致性

**示例**：
- 接口A返回 `orderId` → 接口B使用 `orderId` 查询订单详情
- 接口A返回 `userId` → 接口B使用 `userId` 查询用户信息

### 3. 状态流转场景

**目标**：
- 识别业务状态机（如：订单状态、活动状态等）
- 测试状态的正向流转和逆向流转
- 验证非法状态流转的拦截

**示例状态机**：
```
活动状态流转：
DRAFT（草稿） → APPROVED（已审核） → PUBLISHED（已发布） → CLOSED（已关闭）
```

**测试场景**：
- 正向流转：DRAFT → APPROVED → PUBLISHED → CLOSED
- 非法流转：DRAFT → PUBLISHED（跳过审核，应被拦截）
- 逆向流转：PUBLISHED → DRAFT（已发布回到草稿，视业务规则）

### 4. 并发与时序场景

**目标**：
- 多接口并发调用的一致性
- 接口调用时序依赖测试
- 幂等性验证

**示例**：
- 并发创建：同时调用创建接口，验证数据一致性
- 幂等性：重复调用同一接口，验证结果一致性
- 时序依赖：接口A必须在接口B之前调用

### 5. 异常恢复场景

**目标**：
- 中间步骤失败的回滚测试
- 部分成功场景的数据一致性
- 补偿机制验证

**示例**：
- 订单支付失败后的库存回滚
- 活动审核拒绝后的状态恢复
- 部分成功的批量操作处理

## 场景识别维度

在生成场景测试用例前，需要先分析文档中的接口关系，识别可能的业务场景。

### 1. 接口命名规律识别

通过接口命名识别业务操作类型。

**CRUD 操作识别**：
```
create/add/save → get/query/list → update/modify/edit → delete/remove
```

**状态流转操作识别**：
```
submit → review/audit → approve/reject → publish/close
```

**业务流程操作识别**：
```
register → verify → activate → use → expire/renew
```

**示例**：
```
识别到的接口：
- /activity/create
- /activity/get
- /activity/update
- /activity/delete

场景：CRUD 完整流程
create → get（验证创建结果）→ update → get（验证更新结果）→ delete
```

### 2. 数据依赖关系识别

通过数据流向建立接口关系链。

**识别步骤**：
1. 查找返回 ID 类字段的接口（如：userId, orderId, activityId等）
2. 查找需要 ID 作为入参的接口
3. 建立数据流向关系链

**示例**：
```
接口A：/activity/create
响应：{ "activityId": "ACT123456" }

接口B：/activity/audit
入参：{ "activityId": "..." }

数据依赖：接口A返回的activityId → 接口B的入参
```

### 3. 业务领域聚合

识别同一业务领域的接口，组成业务闭环。

**聚合规则**：
- 同一业务领域的接口（如：活动相关、订单相关、用户相关）
- 识别领域内的完整业务闭环

**示例**：
```
活动领域接口：
- /activity/create
- /activity/audit
- /activity/publish
- /activity/query
- /activity/close

业务闭环：创建 → 审核 → 发布 → 查询 → 关闭
```

### 4. 状态机识别

识别业务状态流转路径。

**识别步骤**：
1. 识别 status、state 类字段
2. 分析状态的可能流转路径
3. 识别状态流转的触发接口

**示例**：
```
状态字段：status
可能值：DRAFT, APPROVED, PUBLISHED, CLOSED

状态流转路径：
DRAFT → APPROVED → PUBLISHED → CLOSED

触发接口：
- /activity/audit → APPROVED
- /activity/publish → PUBLISHED
- /activity/close → CLOSED
```

## 场景提取示例

### 示例 1：活动管理场景

**文档中的接口**：
- `/activity/create` - 创建活动
- `/activity/get` - 查询活动详情
- `/activity/audit` - 审核活动
- `/activity/publish` - 发布活动
- `/activity/close` - 关闭活动

**识别出的场景**：

1. **完整生命周期场景**：
   ```
   create → audit → publish → close
   ```

2. **查询验证场景**：
   ```
   create → get（验证创建结果）
   ```

3. **异常流程场景**：
   ```
   create → audit(reject) → publish（应失败）
   ```

### 示例 2：订单流程场景

**文档中的接口**：
- `/order/create` - 创建订单
- `/order/pay` - 支付订单
- `/order/query` - 查询订单
- `/order/cancel` - 取消订单

**识别出的场景**：

1. **正常下单流程**：
   ```
   create → pay → query（验证支付成功）
   ```

2. **取消订单流程**：
   ```
   create → cancel → query（验证订单已取消）
   ```

3. **异常流程**：
   ```
   create → pay（支付失败）→ query（验证订单状态）
   ```

## 场景用例优先级评估

### 优先级划分标准

| 优先级 | 场景类型 | 特征 | 示例 |
|-------|---------|------|------|
| **P0** | 核心业务流程 | 主流程，影响核心业务 | 用户注册流程、订单支付流程 |
| **P1** | 异常恢复 | 异常处理，影响用户体验 | 支付失败回滚、审核拒绝处理 |
| **P2** | 并发场景 | 高并发下的一致性保障 | 库存扣减并发、积分并发使用 |
| **P3** | 边界场景 | 边界条件处理 | 极限数据量、时间边界 |

### 评估要点

1. **业务重要性**：是否影响核心业务流程
2. **用户影响面**：涉及用户范围大小
3. **风险级别**：失败后的影响程度
4. **使用频率**：场景发生的频率

## 场景用例设计原则

### 1. 真实性

场景应反映真实的业务使用情况。

**示例**：
- ✅ 用户创建活动后立即查询验证（常见操作）
- ❌ 用户创建活动后等待1小时再查询（不常见）

### 2. 完整性

覆盖业务流程的各个环节。

**示例**：
- ✅ 创建 → 审核 → 发布 → 查询 → 关闭（完整流程）
- ❌ 创建 → 发布（跳过审核环节）

### 3. 可验证性

每个步骤都有明确的验证点。

**示例**：
```markdown
验证点：
- ✓ 每个步骤返回的状态码正确
- ✓ 活动状态正确流转（DRAFT → APPROVED → PUBLISHED）
- ✓ activityId 在整个流程中保持一致
- ✓ 最终查询结果与预期一致
```

### 4. 可维护性

场景描述清晰，易于理解和维护。

**示例**：
```markdown
**场景描述**：测试从创建到发布的完整流程
**涉及接口**：
1. 接口A：创建活动 - 返回 activityId
2. 接口B：审核活动 - 使用 activityId
3. 接口C：发布活动 - 使用 activityId
```

### 5. 数据关联

明确标注接口间的数据传递关系。

**示例**：
```markdown
1. 调用接口A创建活动
   - 返回：{ "activityId": "ACT123456" }

2. 使用步骤1返回的activityId，调用接口B审核
   - 请求参数：{ "activityId": "{{步骤1返回的activityId}}" }
```

## 场景识别注意事项

### 1. 场景数量控制

- 优先识别业务主流程，再识别异常分支
- 场景数量不宜过多，聚焦核心业务流程
- **建议**：通常 3-5 个场景即可

### 2. 接口关联性判断

- 关注接口的数据依赖关系
- 不是所有接口都需要组合成场景
- 只有有业务关联的接口才需要设计场景用例

### 3. 单接口 vs 场景用例

- **单接口用例**：聚焦接口自身的功能和契约
- **场景用例**：聚焦业务流程和数据一致性
- 每个场景用例应包含：正向流程 + 至少1个异常流程

### 4. 数据传递标记

使用统一的标记方式标注数据依赖：

```markdown
使用 {{步骤N返回的字段名}} 标记数据依赖
```

**示例**：
```json
{
  "activityId": "{{步骤1返回的activityId}}",
  "userId": "{{步骤2返回的userId}}"
}
```

**好处**：
- 便于测试执行时进行数据提取和传递
- 清晰标识接口间的数据依赖关系
