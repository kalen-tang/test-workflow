# 规范化需求文档格式说明（req-parser 输出）

**版本**: v2.0
**更新时间**: 2026-04-11
**输出工具**: req-parser v2.0

---

## 概述

req-parser 输出的规范化需求文档采用 ZA Bank PRD 模板 7 章结构。case-designer 从中提取功能信息和验收标准来生成测试案例。

## 文档结构

```
# [项目名称] 规范化需求文档

第 1 章 需求背景与目标
  1.1 业务痛点（含数据支撑）
  1.2 可量化目标（含具体指标）

第 2 章 专业术语表
  （表格：名称 | 说明）

第 3 章 业务逻辑与功能需求      ← case-designer 主要提取来源
  3.1 关键业务流程              ← 用于生成 PlantUML 流程图
  3.2 功能需求                  ← 用于识别测试场景
  3.3 界面需求                  ← 用于 UI 测试场景
  3.4 通知需求                  ← 用于通知类测试场景
  3.5 后台管理与CS/OPS需求
  3.6 Finance需求
  3.7 文案

第 4 章 非功能需求
  4.1 性能需求
  4.2 数据与埋点
  4.3 开关控制

第 5 章 验收标准                ← case-designer 核心输入
  （表格：功能点 | 验收标准）
  验收标准格式：给定xxx，当xxx时，那么应xxx

第 6 章 版本规划

第 7 章 附件
```

## case-designer 提取规则

### 从第 3 章提取

1. **3.1 关键业务流程** → 生成 PlantUML Activity Diagram
2. **3.2 功能需求** → 识别功能模块，为每个功能生成测试场景
   - 提取功能名称、适用范围、业务规则
   - 业务规则用于推导边界和异常场景
3. **3.3 界面需求** → 补充 UI 交互的测试场景
4. **3.4 通知需求** → 生成通知触发的测试场景

### 从第 5 章提取

验收标准表格直接作为测试场景的基础：
- **给定（Given）** → 测试前置条件
- **当（When）** → 测试操作步骤
- **那么应（Then）** → 测试预期结果

case-designer 在此基础上扩展：
- 补充负向场景（原文未覆盖的异常路径）
- 补充边界场景（基于业务规则推导）
- 分配测试优先级（P0/P1/P2）

## 注意事项

- 原文标注"原文未涉及"的章节，case-designer 跳过
- 如果第 5 章验收标准为空，case-designer 从第 3 章功能需求自行推导
- 表格格式可能有变体（有些文档用列表而非表格），case-designer 应兼容处理
