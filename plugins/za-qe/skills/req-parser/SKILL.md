# req-parser - 需求文档标准化器

## 📋 概述

将原始需求文档（Word/PDF/Markdown/文本）标准化为结构化的 Markdown 格式，为后续阶段（需求验证、手工案例生成、API 用例生成）提供统一的输入接口。

## 🎯 核心功能

1. **智能提取**：从非结构化文档中提取功能、验收标准、业务规则
2. **繁简转换**：自动识别中文简体/繁体输入，确保输出为简体中文，保持专业术语一致性
3. **测试导向**：标注测试重点（test_focus）、风险等级（risk_level）、关联接口
4. **场景生成**：基于验收标准生成 given-when-then 测试场景
5. **三级检查**：P0/P1/P2 三级一致性检查，确保输出质量
6. **置信度标记**：对提取内容标注置信度（high/medium/low），便于人工复核
7. **结构化识别**：自动识别文档中的表格、列表、章节结构元素，提取场景和说明

## 📥 输入

- **需求文档**：Word（.docx）、PDF、Markdown（.md）、纯文本（.txt）
- **可选配置**：
  - `module_name`：模块名称（如未提供，从文档标题推断）
  - `strict_mode`：严格模式（P0 检查失败时阻塞，默认 true）

## 📤 输出

- **主产出物**：标准化需求文档（Markdown 格式）
  - 符合 [normalized-requirement.md](references/normalized-requirement.md) 规范
  - 包含元数据、功能列表、验收标准、测试场景、业务规则
  - 包含三级一致性检查报告

- **可选产出物**：
  - 缺失项清单（missing_items）
  - 改进建议（recommendations）

## 🚀 使用方式

### 命令行使用

```bash
# 基本用法
/req-parser ./docs/需求文档V1.0.docx

# 指定模块名称
/req-parser ./docs/需求文档.docx --module "用户管理模块"

# 非严格模式（P0 检查失败仍继续）
/req-parser ./docs/需求文档.docx --strict false
```

### 在 /za-qe:full-workflow 工作流中使用

```bash
# 完整模式会自动调用 req-parser
/za-qe:full-workflow ./project-root
```

## 🔧 处理流程

### 第一步：文档解析

1. **识别文档格式**：
   - Word（.docx）→ 使用 python-docx 解析
   - PDF → 使用 PyPDF2 提取文本
   - Markdown → 直接解析
   - 纯文本 → 直接读取

2. **提取章节结构**：
   - 识别标题层级（H1/H2/H3）
   - 构建文档大纲

3. **结构化内容提取**：
   - 如果文档中存在图片，一并分析；无法解析时保留原始内容
   - 如果需求内容存在表格，提取表格中的【场景】、【说明】部分，作为需求内容继续分析
   - 如果文档存在附件或超链接，保留原始内容，作为输出的附件章节

4. **繁简转换**：
   - 自动识别中文繁体输入，统一转换为简体中文输出
   - 保持技术术语和专业名词的一致性，避免翻译失真

### 第二步：信息提取

#### 2.1 元数据提取

```markdown
<!-- 从文档中提取 -->
- **模块**：用户管理模块（从标题或首段推断）
- **文档版本**：V1.2（从文档版本号提取）
- **处理时间**：2026-03-18T10:30:00+08:00
- **处理工具**：req-parser v1.0
```

#### 2.2 功能提取

**识别模式**：
- 标题包含"功能"、"需求"、"Feature" → 识别为功能章节
- 编号列表（1.x、2.x）→ 识别为功能点

**提取字段**：
```markdown
### F001 用户登录
- **描述**：...（从段落文本提取）
- **优先级**：high（从关键词推断：重要/核心 → high）
- **测试重点**：是（从优先级 + 风险级别推断）
- **风险等级**：medium（从关键词推断：资金/安全 → high）
- **关联接口**：（从文本中识别接口路径 /xxx/xxx）
- **来源章节**：需求文档 3.1 节
- **置信度**：high
```

**置信度判断**：
- `high`：原文明确写出功能名称和描述
- `medium`：从章节标题推断，描述需要总结
- `low`：完全从上下文推测

#### 2.3 验收标准提取

**识别模式**：
- 段落包含"验收标准"、"Acceptance Criteria"
- 段落包含"应该"、"必须"、"能够"
- 列表形式的条件描述

**提取字段**：
```markdown
**验收标准**：
- AC001（关联：F001，来源：需求文档 3.1.1，置信度：high）：用户输入正确的手机号和密码，点击登录按钮，跳转到首页
```

#### 2.4 业务规则提取

**识别模式**：
- 段落包含"规则"、"Rule"、"约束"、"Constraint"
- 段落包含"必须"、"不能"、"应该"
- 格式化的约束描述（如"XX长度为XX"）

**分类规则**：
```markdown
**业务规则**：
- BR001（类型：data_rule，来源：需求文档 4.1，置信度：high，关联：F001 F002）：手机号必须为11位数字
```

**类型判断**：
- `data_rule`：字段格式、长度、范围约束
- `calculation_rule`：计算公式、逻辑规则
- `process_rule`：流程控制、状态转换
- `permission_rule`：权限、角色限制

#### 2.5 测试场景生成

**生成策略**：

1. **从验收标准生成正向场景**：
   ```markdown
   <!-- 验收标准：用户输入正确的手机号和密码，点击登录按钮，跳转到首页 -->
   ### S001 正常登录-正确账号密码（positive）
   - **关联**：AC001
   - **Given**：用户已注册，手机号为13800138000，密码为Test1234
   - **When**：用户输入手机号13800138000，密码Test1234，点击登录
   - **Then**：跳转到首页，显示用户昵称
   - **来源**：requirement
   - **置信度**：high
   ```

2. **从业务规则生成边界场景**：
   ```markdown
   <!-- 业务规则：手机号必须为11位数字 -->
   ### S004 边界-手机号格式错误（boundary）
   - **关联**：BR001
   - **Given**：用户在登录页
   - **When**：用户输入手机号12345678（不足11位），点击登录
   - **Then**：显示提示'手机号格式错误'
   - **来源**：supplement
   - **置信度**：medium
   ```

3. **从验收标准推断负向场景**：
   ```markdown
   <!-- 验收标准：用户输入错误的密码，显示错误提示 -->
   ### S002 异常登录-错误密码（negative）
   - **关联**：AC002
   - **Given**：用户已注册，手机号为13800138000，正确密码为Test1234
   - **When**：用户输入手机号13800138000，密码WrongPass，点击登录
   - **Then**：停留在登录页，显示错误提示'密码错误'
   - **来源**：requirement
   - **置信度**：high
   ```

#### 2.6 缺失项识别

**检查维度**：
- 功能是否有明确的验收标准？
- 验收标准是否有对应的测试场景？
- 业务规则是否有验证场景？
- 字段约束是否完整？
- 关联接口是否明确？

**记录方式**：
```markdown
## 缺失项清单

- [ ] 未明确说明账号锁定30分钟后是否自动解锁
- [ ] 未说明密码输入框是否需要显示/隐藏切换
- [ ] 未说明是否支持第三方登录（微信/支付宝）
```

### 第三步：一致性检查

#### P0 级检查（必须通过）

```markdown
### P0 检查（必须通过）

| 检查项 | 规则 | 结果 | 失败详情 |
|--------|------|------|---------|
| 功能与验收标准对应 | 每个功能至少有一个验收标准关联 | ✅ pass | — |
| 场景内部逻辑完整 | 每个场景的given-when-then不能有逻辑矛盾 | ✅ pass | — |
| 字段约束一致性 | 字段定义与业务规则不能冲突 | ✅ pass | — |
```

**检查逻辑**：
1. 遍历 features，检查是否有 relates_to 该功能的 acceptance_criteria
2. 遍历 scenarios，检查 given-when-then 的逻辑关系
3. 遍历 fields，检查 rules 引用的 business_rules 是否存在且不冲突

**失败处理**：
- 如果 `strict_mode=true`（默认）：返回错误报告，不生成输出文件
- 如果 `strict_mode=false`：记录失败项，继续生成

#### P1 级检查（必须记录）

```markdown
### P1 检查（必须记录）

| 检查项 | 规则 | 结果 | 缺失项 |
|--------|------|------|--------|
| 验收标准与场景对应 | 每个验收标准至少有一个对应的测试场景 | partial | AC003 |
| 业务规则与场景对应 | 每个业务规则至少在一个场景中被验证 | partial | BR002 |
| 必填字段测试覆盖 | 每个必填字段，应有'不填'的负向场景 | partial | password |
```

**检查逻辑**：
1. 遍历 acceptance_criteria，检查是否有 relates_to 该 AC 的 scenario
2. 遍历 business_rules，检查是否在任何 scenario 中被提及或验证
3. 遍历 fields（required=true），检查是否有对应的负向/边界场景

**结果类型**：
- `pass`：100% 覆盖
- `partial`：部分覆盖（50%-99%）
- `fail`：覆盖率 < 50%

#### P2 级检查（建议项）

```markdown
### P2 检查（建议项）

| 检查项 | 规则 | 结果 | 说明 |
|--------|------|------|------|
| 正负场景比例 | 负向场景数量应不少于正向场景数量 | ✅ pass | 正向1 / 负向2 / 边界1 |
| 缺失项记录 | 所有必须但缺失的内容已在缺失项清单中记录 | ✅ pass | — |
```

### 第四步：生成改进建议

基于 P1/P2 检查结果，生成改进建议：

```markdown
## 改进建议

| 优先级 | 建议内容 | 原因 |
|--------|---------|------|
| high | 补充业务规则 BR002（密码长度）的验证场景 | P1 检查发现该业务规则未被场景覆盖 |
| medium | 增加 password 字段的'不填'负向场景 | 必填字段应有对应的边界场景 |
| low | 明确账号锁定后的解锁机制（自动/手动） | 缺失项清单中记录的缺失内容 |
```

## 📊 质量保证

### 输出质量指标

| 指标 | 目标 | 说明 |
|------|------|------|
| P0 检查通过率 | 100% | 严格模式下必须 100%，否则不输出 |
| 功能提取准确率 | > 95% | 与人工标注对比 |
| 验收标准提取准确率 | > 90% | 与人工标注对比 |
| 场景生成覆盖率 | > 80% | 验收标准→场景的覆盖率 |
| 置信度标记准确率 | > 85% | high 标记的准确率 |

### 人工复核建议

1. **优先复核低置信度内容**：
   - 查找所有 `confidence: medium` 或 `confidence: low` 的字段
   - 对照原文验证是否正确

2. **检查缺失项**：
   - 查看 `missing_items` 列表
   - 补充缺失内容或确认不需要

3. **验证改进建议**：
   - 查看 `recommendations` 列表
   - 根据优先级补充场景或规则

## ⚠️ 注意事项

1. **文档质量影响输出质量**：
   - 如果原始需求文档结构混乱、表达不清，提取质量会下降
   - 建议在使用前，确保需求文档有基本的章节结构

2. **置信度标记需要人工复核**：
   - `confidence: medium` 或 `low` 的内容必须人工确认
   - 特别是 `test_focus` 和 `risk_level` 的判断

3. **关联接口可能不准确**：
   - 如果需求文档未明确接口路径，只能推断
   - 建议结合设计文档（02-normalized-design.md）进行二次校验

4. **严格模式建议开启**：
   - 默认 `strict_mode=true`，P0 检查失败时会阻塞
   - 这确保输出质量，避免垃圾输入影响后续阶段

5. **支持的文档格式有限**：
   - Word（.docx）支持最好
   - PDF 仅支持文本提取（图片中的文字无法识别）
   - 图片格式（如 PNG、JPG）不支持

## 🚧 已知限制

1. **不支持表格解析**：当前版本不支持从表格中提取结构化信息
2. **不支持图片识别**：需求文档中的流程图、界面图无法识别
3. **不支持多语言**：仅支持中文和英文需求文档
4. **业务规则分类准确率有限**：`type` 字段的判断可能不准确，建议人工复核

## 🏢 应用场景

- 自动识别需求文档中的缺失项和逻辑不一致
- 使用指令：`请分析这份PRD文档并提取所有功能点的业务规则和技术约束`

- 基于验收标准生成可执行的测试场景（given-when-then）
- 从业务规则自动推导边界场景
- 使用指令：`检查需求文档的完整性和可测试性，标注缺失的技术实现细节`

- 确保团队需求文档格式和内容的一致性
