# req-parser - 需求文档标准化器

## 概述

将原始需求文档标准化为 ZA Bank PRD 模板的 7 章结构，为后续 case-designer、api-generator 等 Skill 提供统一输入。

**核心原则**：以 PRD 模板（`references/prd-template.md`）为标准，最小颗粒度符合即可。不做过度推断，原文没有的章节留空或标注"原文未涉及"。

## 输入

- **需求文档**：Markdown（.md）、Word（.docx）、纯文本（.txt）
- **可选参数**：
  - `module_name`：模块名称（未提供则从文档标题推断）

## 输出

**主产出物**：`<案例输出目录>/<模块名>_规范化需求文档.md`

输出结构严格对齐 PRD 模板 7 章：

```
# [项目名称] 规范化需求文档

文档修改历史记录（从原文提取）

第 1 章 需求背景与目标
  1.1 业务痛点
  1.2 可量化目标

第 2 章 专业术语表

第 3 章 业务逻辑与功能需求
  3.1 关键业务流程
  3.2 功能需求
  3.3 界面需求
  3.4 通知需求
  3.5 后台管理与CS/OPS需求
  3.6 Finance需求
  3.7 文案

第 4 章 非功能需求
  4.1 性能需求
  4.2 数据与埋点
  4.3 开关控制

第 5 章 验收标准

第 6 章 版本规划

第 7 章 附件
```

## 处理流程

### 第一步：文档解析与复杂度评估

1. **读取文档**：使用 Read 工具读取输入文件
2. **繁简转换**：自动识别繁体中文，统一转换为简体中文输出
3. **结构识别**：
   - 识别标题层级（H1/H2/H3）
   - 识别表格、列表、图片引用
   - 提取章节结构和内容
4. **复杂度评估**：统计功能模块数量

```
复杂度评估：{N} 个功能模块 → 使用{直接/分段}生成模式
```

**分段决策**：
- 功能模块 ≤ 3 个 → **直接生成模式**（一次性 Write 完整文件）
- 功能模块 > 3 个 → **分段生成模式**（Write 骨架 + Edit 逐段追加）

### 第二步：信息提取与映射

将原文内容映射到 7 章结构。提取规则：

#### 第 1 章：需求背景与目标

**提取来源**：原文中包含"背景"、"目标"、"痛点"、"为什么"、"业务价值"等关键词的章节

- **业务痛点**：需有数据支撑。原文有数据直接引用；无数据则保留原文描述
- **可量化目标**：提取具体数值指标。原文无量化指标时标注"原文未提供量化指标"

#### 第 2 章：专业术语表

**提取来源**：原文中的术语定义、名词解释、缩写说明

| 名称 | 说明 |
|------|------|

原文无术语定义时，提取文档中出现的专业术语或缩写，补充常见含义。

#### 第 3 章：业务逻辑与功能需求

**提取来源**：原文中的功能描述、业务流程、界面设计、通知规则、后台管理等

按以下子章节组织：

- **3.1 关键业务流程**：从原文提取流程图描述或步骤列表
- **3.2 功能需求**：逐条提取功能描述，保留原文结构
  - 每个功能包含：适用范围、功能描述、业务规则
  - 如涉及表格结构，保留原始表格
- **3.3 界面需求**：场景 + 说明 + 截图引用
- **3.4 通知需求**：场景 + 渠道（Push/Email/SMS）+ 备注
- **3.5 后台管理与CS/OPS需求**：后台配置、运营参数
- **3.6 Finance需求**：收费、GL、记账对账（如涉及）
- **3.7 文案**：简体/繁体/英文三语对照（如涉及）

**重要**：
- 原文有的内容如实提取，不做臆造
- 原文的表格、列表格式尽量保留
- 原文涉及图片的，保留图片引用标记

#### 第 4 章：非功能需求

**提取来源**：性能指标、埋点方案、开关配置、灰度策略、版本兼容等

- **4.1 性能需求**：响应时间、吞吐量等指标
- **4.2 数据与埋点**：埋点页面、事件名称
- **4.3 开关控制**：开关编码、名称、默认状态、灰度策略

原文未涉及的子章节标注"原文未涉及"。

#### 第 5 章：验收标准

**提取来源**：原文中的验收标准、验收条件、测试通过条件

**格式要求**（给定-当-则）：

| **功能点** | **验收标准** |
|---|---|
| 功能A | 1. 给定xxx，当xxx时，那么应xxx 2. 给定xxx，当xxx时，那么应xxx |
| 功能B | 1. 指标名≥指标值 |

**生成规则**：
- 原文已有验收标准的，直接按上述格式整理
- 原文无验收标准但有功能描述的，从功能描述中推导基本验收标准
- 推导的验收标准仅覆盖正向主流程，不展开边界和异常场景（留给 case-designer）

#### 第 6 章：版本规划

**提取来源**：原文中的版本计划、迭代规划、里程碑

| **版本** | **时间节点** | **核心迭代内容** |
|---|---|---|

原文无版本规划时标注"原文未涉及"。

#### 第 7 章：附件

**提取来源**：原文中引用的外部链接、设计稿、流程图文件

保留原文的链接和引用。

### 第三步：输出文件

#### 直接生成模式（功能模块 ≤ 3）

使用 Write 工具一次性写出完整的 7 章文档。

#### 分段生成模式（功能模块 > 3）

**Step 1**：使用 Write 工具写出骨架文件，包含：
- 文件头（标题 + 元信息）
- 第 1 章、第 2 章完整内容
- 第 3 章标题 + 各功能模块标题（无详细内容）
- 第 4-7 章占位符：`<!-- PLACEHOLDER: 待生成 -->`

**Step 2**：将功能模块按每组 2-3 个分组，逐组处理：
1. Read 当前文件，确认插入位置
2. 为该组功能生成第 3 章详细内容（功能需求、界面需求等）
3. 使用 Edit 工具追加到文件对应位置

每组完成后输出进度：
```
分段进度：第 {n}/{total} 组完成
```

**Step 3**：Read 完整文件，使用 Edit 替换占位符，生成：
- 第 4 章 非功能需求
- 第 5 章 验收标准（基于已生成的所有功能）
- 第 6 章 版本规划
- 第 7 章 附件

**Step 4**：Read 最终文件，确认无残留占位符。

### 第四步：输出摘要

```
规范化完成：
  模块名称：xxx
  功能模块：N 个
  验收标准：M 条
  生成模式：直接/分段
  输出文件：<路径>
```

## 使用方式

```bash
# 基本用法
/req-parser ./docs/需求文档.md

# 指定模块名称
/req-parser ./docs/需求文档.md --module "用户管理模块"
```

## 与其他 Skill 对接

### 输出给 case-designer

case-designer 读取本 Skill 的输出文件，从中提取：
- 第 3 章的功能需求和业务流程 → 生成 PlantUML 流程图和测试案例
- 第 5 章的验收标准 → 作为测试场景的基础

### 输出给 api-generator

api-generator 读取本 Skill 的输出文件，从中提取：
- 第 3 章的功能需求和接口信息 → 生成 API 测试代码
- 第 5 章的验收标准 → 转换为自动化断言

## 注意事项

1. **忠于原文**：原文有的内容如实提取整理，原文没有的不做臆造，标注"原文未涉及"
2. **繁简转换**：自动识别繁体中文输入，统一输出简体中文
3. **表格保留**：原文中的表格尽量保留原始结构
4. **图片标记**：原文中的图片保留引用标记（如 `![描述](图片)`）
5. **分段生成**：大文档（>3 功能模块）自动切换分段模式，避免超时
