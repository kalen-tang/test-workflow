# 标准开发方案文档规范

本文档定义银行开发方案文档的规范结构，用于 design-parser 技能的检查基准。

---

## 文档整体结构

标准开发方案文档包含以下章节，分为**必须章节**和**按需章节**两类：

```
标准开发方案文档
├── [必须] 2. 整体概要设计
│   ├── 2.1 整体架构概览
│   └── 2.2 配置类信息
├── [必须] 4. 需求模块拆解（含任务分工表）
├── [必须] 5. 分模块概要设计（每个模块）
│   ├── 5.x.1 流程图（按需）
│   ├── 5.x.2 UML（按需）
│   ├── 5.x.3 数据库变更（按需）
│   ├── [必须] 5.x.4 接口报文
│   └── 5.x.5 关联逻辑兼容性分析（按需）
├── [按需] 1. 需求背景描述
├── [按需] 3. 架构变更及评审
├── [按需] 6. 整体兼容性分析
├── [按需] 7. 兼容性方案
├── [按需] 8. 业务降级和数据一致性梳理
├── [按需] 9. 多线程安全梳理
├── [按需] 10. 发布物料
├── [按需] 11. 测试重点关注要点列举
├── [按需] 12. 发布后生产配置清单
└── [按需] 13. 下线配置清单
```

---

## 必须章节规范

### 第2章：整体概要设计（必须）

**2.1 整体架构概览**

- **要求**：描述基本架构概览，可引用外部架构文档
- **合格标准**：有实质性内容（非空、非占位符）
- **提示**：可以写"见外部文档链接"形式，但不能完全为空

**2.2 配置类信息**

- **要求**：列出配置项和开发类信息
- **合格标准**：有内容，或明确标注"无配置变更"
- **提示**：若无配置变更，可写"无"

---

### 第4章：需求模块拆解（必须）

**要求**：

1. 列出所有需求模块（功能模块列表）
2. 包含**任务分工表**，字段：模块名 | 后端开发人员 | 测试人员

**合格标准**：
- 至少列出一个模块
- 每个模块需有对应的第5章内容

**示例格式**：
```
| 模块       | 后端开发 | 测试    |
|------------|---------|---------|
| 助力膨胀   | 张三     | 李四    |
| 限时秒杀   | 王五     | 赵六    |
| 商户专区   | 钱七     | 孙八    |
```

---

### 第5章：分模块概要设计（必须）

每个在第4章中声明的模块**必须**有对应的第5.x节，每节结构如下：

#### 5.x.4 接口报文（必须）

接口信息是最核心的必须内容，每个接口需包含：

| 字段 | 是否必须 | 说明 |
|------|---------|------|
| 接口名称/编号 | 必须 | 如"1.消息订阅查询" |
| UDOC 文档链接 | 推荐 | 如 `https://udoc.in.za/#/view/xxx` |
| 接口 URL | 必须 | 如 `/dmb/ncjr02/userPush/subscribe/status/get` |
| 入参字段列表 | 必须 | 含字段名、类型、是否必填、字段说明 |
| 出参字段列表 | 必须 | 含字段名、类型、是否必填、字段说明 |
| 备注说明 | 按需 | 特殊逻辑、降级策略等 |

**接口 URL 格式规范**：
- 微服务接口：`微服务域名/接口路径`，如 `zabank_imc_activity_service/activity/list`
- 网关接口：路径含 `/dmb/`，如 `/dmb/ncjr02/userPush/subscribe`
- **注意**：网关接口（含 dmb）需提示测试人员确认是否有对应微服务接口

**接口合格标准**：
- 接口 URL 不为空
- 入参/出参至少有字段列表（即使只有基础字段 projectId/mainActivityId）
- 若出参为空，需明确说明原因

#### 5.x.1 流程图（按需）

- 可以写"无"
- 若有流程图，推荐使用 PlantUML 或图片引用
- 重要模块（核心业务流）建议提供

#### 5.x.2 UML（按需）

- 可以写"无"
- 包括时序图、类图、状态图等

#### 5.x.3 数据库变更（按需）

- 可以写"无"
- 若有变更，需提供完整 DDL/SQL
- 表结构需含完整字段注释（COMMENT）

#### 5.x.5 关联逻辑兼容性分析（按需）

- 描述新功能对现有功能的影响
- 可以写"新功能无关联影响"

---

## 按需章节规范

### 第1章：需求背景描述（按需）

包含：
- 1.1 背景：业务背景说明
- 1.2 功能简介&业务效果：可引用 PRD 文档地址

### 第3章：架构变更及评审（按需）

- 若无变更，写"无架构变更"
- 若有变更，需说明变更内容和评审结论

### 第6章：整体兼容性分析（按需）

包含六类分析：
1. 上游调用方分析（6.1）
2. 下游被调用方分析（6.2）
3. App 兼容性分析（6.3）：老版本/新版本 App 与新老代码/数据的兼容
4. 数据兼容性分析（6.4）
5. 后管兼容性分析（6.5）
6. 发布机房兼容性分析（6.6）：多版本 Pod 间兼容、多机房切流、发布时序

**要求**：若存在此章节，每个子项都要有分析结论，不能留空。

### 第7章：兼容性方案（按需）

列出具体的兼容方案，与第6章配套。

### 第8章：业务降级和数据一致性梳理（按需）

以表格形式列出每个功能点：

| 功能点 | 调用下游 | 是否启用熔断/降级 | 不启用的原因 | 业务降级方案 | 是否存在多发数据一致性场景 | 数据一致性方案 |
|--------|---------|-----------------|-------------|------------|--------------------------|--------------|

**8.1 限流熔断配置**：若有接口需限流，提供 Sentinel 配置（流控规则 + 降级规则）。

### 第9章：多线程安全梳理（按需）

列出并发场景和处理方案。

### 第10章：发布物料（按需）

包含：
- 10.1 发布系统：服务列表、发布方式、Tag
- 10.2 MySQL 物料：DDL/DML SQL
- 10.3 定时任务物料：PowerJob 任务配置 JSON
- 10.4 Flink 物料（如适用）
- 10.5 HBase 表结构（如适用）
- 10.6 Apollo 配置：新增/修改/删除的配置项
- 10.7 CK 表结构物料（如适用）
- 10.8 生产初始化物料

### 第11章：测试重点关注要点列举（按需）

开发侧提示测试人员需重点关注的测试点。

### 第12章：发布后生产配置清单（按需）

发布上线后需要执行的配置操作。

### 第13章：下线配置清单（按需）

需要下线的配置项列表。

## 常见问题和扣分项

### 高优先级问题（严重缺失）

| 问题 | 说明 |
|------|------|
| 缺少整体概要设计（第2章） | 无法了解系统架构 |
| 缺少需求模块拆解（第4章） | 无法分工和追踪 |
| 缺少某模块的接口信息（5.x.4） | 测试无法进行接口测试 |
| 接口 URL 为空 | 测试无法定位接口 |
| 接口入参/出参缺失 | 无法构造测试数据 |

### 中优先级问题（不完整）

| 问题 | 说明 |
|------|------|
| 接口使用网关地址（含 /dmb/）但未标注微服务接口 | 测试需确认真实接口路径 |
| 第4章模块在第5章中无对应内容 | 设计不完整 |
| UDOC 链接为空（只有占位符）| 影响测试文档查阅 |
| 兼容性分析章节存在但子项为空 | 风险评估不完整 |

### 低优先级问题（建议改进）

| 问题 | 说明 |
|------|------|
| 流程图/UML 缺失（未写"无"） | 格式不规范 |
| 数据库字段缺少 COMMENT | 可维护性差 |
| 接口出参字段不完整 | 测试用例设计依赖度高 |
| 第10章发布物料不完整 | 影响发布执行 |
