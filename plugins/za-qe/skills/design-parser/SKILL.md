# 开发方案文档规范化器

## 技能目标

读取原始开发方案文档，按照团队规范进行格式化和补全，产出一份**规范化的 Markdown 文档**，供 `api-generator` 技能消费生成接口自动化测试。

## 必须章节 vs 按需章节

规范详细定义见 `references/devplan-standard.md`。核心要点：

**必须章节**：第2章（整体概要设计）、第4章（需求模块拆解）、第5章每个模块的 5.x.4 接口报文

**按需章节**：流程图、UML、数据库变更、兼容性分析等，原文有则保留，原文没有则标注"无"

## 执行流程

### 第一步：读取原始文档

接受用户提供的文档路径（支持 .md、.docx、.pdf），读取全文内容。

**不同格式的读取方式**：

| 格式 | 读取方式 |
|------|---------|
| `.md` | 直接使用 Read 工具读取 |
| `.docx` | 用下方的 Python 脚本一次性提取全部文本和表格，写入临时文件后用 Read 工具读取 |
| `.pdf` | 使用 Read 工具直接读取（支持 PDF） |

**.docx 标准读取脚本**：见 `references/docx-reader.md`，将 `YOUR_DOC_PATH` 替换为实际路径后直接运行。运行后用 Read 工具读取 `result/_tmp_docx.txt`，完成后删除该临时文件。

读取内容后：
- 识别章节编号和标题
- 提取所有接口信息（URL、入参、出参、UDOC 链接）
- 提取第4章模块列表，用于对照第5章完整性

### 第二步：提取 UDOC 链接中的接口数据

**在检查内容之前，先处理文档中出现的所有 UDOC 链接。**

#### UDOC 链接识别

扫描全文，找出所有符合以下格式的 UDOC 链接：

```
https://udoc.in.za/#/view/xxxxx
```

对每个识别到的 UDOC 链接，按照 `references/udoc-fetcher.md` 中的方式访问并提取完整接口数据。

#### UDOC 数据提取流程

详细步骤见 `references/udoc-fetcher.md`，核心要点如下：

1. **尝试直接访问**：使用 WebFetch 工具访问该链接
2. **若需要登录**：使用账号 `admin` / 密码 `Za123456` 登录后再访问（见参考文档中的登录方案）
3. **提取以下字段**：

| 字段 | 说明 |
|------|------|
| 接口所属微服务 | 服务名，如 `zabank_imc_activity_service` |
| 接口路径 | 如 `/activity/list` |
| 请求方法 | GET / POST / PUT / DELETE |
| Content-Type | 如 `application/json`、`application/x-www-form-urlencoded`，UDOC 未提供时根据请求方法推断默认值 |
| 功能描述 | 接口说明文字 |
| 请求参数 | 字段名、类型、是否必填、说明（完整列表） |
| 响应参数 | 字段名、类型、说明（完整列表，含嵌套结构） |

4. **用提取到的数据替换/补全**原文该接口的对应信息：
   - 若原文接口信息不完整（入参/出参缺失或字段不全），**以 UDOC 数据为准**覆盖补全
   - 若原文与 UDOC 存在冲突（如 URL 不同），以UDOC为准 
   - 若 UDOC 链接无法访问（网络不通、链接失效等），在接口下方标注 `> ⚠️ UDOC 链接无法访问，接口信息需人工确认`

### 第三步：检查并补全内容

**对缺失或不规范的内容进行处理**：

| 情况 | 处理方式 |
|------|---------|
| 按需章节（流程图/UML/DB变更等）原文为空或未提及 | 补充该小节标题并标注"无" |
| 接口入参/出参表格缺失字段（序号/类型/必填/说明） | 按原文内容补全表格列，缺失列用"—"占位 |
| 接口 URL 为网关地址（含 `/dmb/`） | 在该接口下方添加说明：`> ⚠️ 网关接口，请确认对应微服务接口地址` |
| 必须章节（第2章/第4章/5.x.4）原文完全缺失 | **不自动补全**，在对应位置插入 `> ❌ [缺失] 请补充：xxx` 提示，并在文档末尾汇总问题清单 |

**不改变原文内容**：只做格式规范化和缺失标注，不修改、不推测原文的业务内容。

### 第四步：生成规范化 MD 文档

输出文件保存至 `./result/[需求ID]_规范化开发方案.md`，文件结构如下：

```markdown
# [需求ID] [需求名称] 开发方案

> 本文档由 design-parser 从原始开发方案规范化生成，供接口自动化测试使用。
> 原始文档：[原文档路径]

---

[原文内容]

[原文内容，若原文无则标注"无"]
```

### 第五步：告知用户结果

输出完成后，简洁告知：
1. 文件保存路径
2. 若有"待补充清单"，列出缺失项，提示用户补充后重新执行
3. 若无问题，提示可直接执行 `api-generator` 消费该文件
