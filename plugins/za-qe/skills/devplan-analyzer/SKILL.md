# 测试左移接口分析与测试用例推荐

## 技能目标

专门用于测试左移的前置工作，通过分析开发方案文档（必须）和需求文档（可选），提取关键接口信息，识别接口间的业务关联，生成单接口测试用例和场景测试用例。**当同时提供需求文档时，可生成更全面、更贴近业务场景的测试用例**，帮助测试团队在开发早期阶段介入，全面覆盖接口功能和业务流程。

### 文档输入说明

1. **开发方案文档**（必须）：
   - 提供接口技术规格（接口路径、请求参数、响应参数）
   - 提供接口间的技术调用关系
   - 提供技术实现细节

2. **需求文档**（可选，强烈推荐）：
   - 提供完整的业务流程和业务场景描述
   - 提供用户故事和验收标准
   - 提供业务规则和边界条件
   - **当同时提供时，场景测试用例会更加全面和贴近实际业务**

3. **标准化需求文档**（可选）：
   - 如果用户已使用 `req-parser` 生成了标准化需求文档（YAML 格式），可直接读取
   - 格式参见：`../../references/artifact-schemas/01-normalized-requirement.md`

### 使用建议

- **仅有开发方案时**：可生成接口级别的测试用例和基础场景用例（基于接口命名和数据依赖推断）
- **开发方案 + 需求文档**：可生成更完整的场景用例，覆盖用户故事、验收标准、异常流程等
- **最佳实践**：同时提供开发方案文档和需求文档，确保测试用例既技术准确又业务完整

## 核心功能

### 1. 网页内容提取

若需要从网页中获取内容，使用 Playwright 从提供的链接中提取网页内容（标题、正文、图片等）。

**Playwright MCP 安装**（如未安装）：
```bash
claude mcp add playwright npx @playwright/mcp@latest
```

**UDoc 登录信息**（域名为 `udoc.in.za`）：
- 账号：`admin`
- 密码：`Za123456`

如果网页需要其他登录凭证，提示用户提供。

### 2. 需求文档分析（可选，推荐）

如果用户提供了需求文档，首先分析需求文档以获取业务上下文：

#### 2.1 原始需求文档分析

如果是 Word/PDF/Markdown 格式的需求文档：
- 提取业务流程描述
- 识别用户故事（As a ... I want ... So that ...）
- 提取验收标准（Given ... When ... Then ...）
- 识别业务规则和约束条件
- 提取异常场景和边界条件

#### 2.2 标准化需求文档分析

如果是已标准化的 YAML 格式需求文档（`normalized_requirement`）：
- 直接读取 `features` 字段获取功能列表
- 直接读取 `user_stories` 获取用户故事
- 直接读取 `acceptance_criteria` 获取验收标准
- 直接读取 `business_rules` 获取业务规则
- 直接读取 `scenarios` 获取测试场景模板

#### 2.3 业务场景提取

从需求文档中提取以下信息用于场景用例生成：
- **主要业务流程**：完整的端到端用户操作流程
- **用户角色**：不同角色的操作权限和行为差异
- **验收条件**：每个功能的成功标准
- **异常场景**：业务规则约束、错误处理、边界条件
- **业务状态流转**：业务对象的状态变化路径

> 📖 **详细指南**：参见 `references/05-requirement-integration.md` - 需求文档与开发方案文档结合分析指南

### 3. 开发方案文档分析

分析开发方案文档时：
- 全面解析文档结构和内容
- 识别技术架构信息
- 提取业务需求和功能描述
- 分析系统设计方案
- **检查文档质量**：
  - 开发方案文档应包含完整的接口设计信息（直接列出接口定义或提供 UDoc 链接）
  - 建议使用标准的 API 文档格式（如 OpenAPI/Swagger）
  - 接口参数说明应详细完整
  - 包含接口错误码信息

如果文档不符合规范，提示用户完善文档信息。

### 4. 接口信息提取

从开发方案文档中提取接口信息，确保入参、出参和接口对应正确。

**提取来源**：
1. **直接提取**：如果开发文档直接列出了接口定义，则直接提取
2. **从 UDoc 链接提取**：如果文档中只附上了接口链接（域名为 `udoc.in.za`），使用 Playwright 从链接中提取接口定义

**基础接口信息**：
- **接口名称**：API 接口的业务名称
- **接口路径**：RESTful API 的 URL 路径（需包含微服务域名+接口路径）
- **请求方法**：HTTP 方法（GET、POST、PUT、DELETE等）
- **功能描述**：接口实现的业务功能

**详细参数信息**：
- **请求参数**：路径参数、查询参数、请求头参数、请求体参数、参数类型、是否必填、默认值、取值范围
- **响应参数**：响应状态码、响应头信息、响应体结构、数据类型和字段说明、错误响应格式

**接口路径校验**：
提取接口路径后，必须进行以下校验：

1. **网关接口检测**：
   - 如果接口路径中包含 `dmb`，则判定为网关接口
   - **必须提示用户**：检测到网关接口，测试左移需要使用微服务接口，请替换为对应的微服务接口路径
   - 停止该接口的后续分析，等待用户提供微服务接口

2. **微服务识别**：
   - 如果路径中不包含 `dmb`，则根据路径前缀自动识别所属微服务
   - 微服务映射规则参见 `references/01-interface-validation.md`

3. **完整路径格式**：
   - 标准格式：`微服务域名 + 接口路径`
   - 示例：`zabank_imc_activity_service/activity/list`
   - 如果文档中只提供了接口路径，需要根据映射规则补充微服务域名

> 📖 **详细规则**：参见 `references/01-interface-validation.md` - 接口路径校验与微服务映射规则

### 5. 测试用例推荐

**重要提示**：场景测试用例的生成策略取决于是否提供需求文档：

- **仅有开发方案**：基于接口命名规律、数据依赖关系生成基础场景用例
- **有需求文档**：结合需求文档中的用户故事、验收标准、业务规则生成全面的场景用例

#### 单接口测试用例

为每个接口设计以下类型的测试用例：
- **正常流程测试**：验证接口正常业务逻辑
- **边界值测试**：参数边界值、极限值测试
- **异常流程测试**：错误参数、异常场景测试
- **业务规则验证**：业务逻辑正确性验证

**接口契约测试**：
- 请求参数校验（必填参数、参数格式、参数类型）
- 响应格式验证（响应结构、字段类型、数据格式）
- 状态码验证（各种场景下的 HTTP 状态码）
- 错误处理验证（异常情况下的错误响应格式）

> 📖 **详细模式**：参见 `references/04-test-case-patterns.md` - 测试用例设计模式

#### 场景测试用例

基于提取的多个接口，分析业务流程和数据依赖关系，设计跨接口的场景测试用例。**当提供需求文档时，场景用例会更加全面和准确。**

**场景类型**：
- **业务流程场景**：识别完整的业务流程（如：创建 → 查询 → 更新 → 删除），设计端到端的业务场景测试
  - *有需求文档*：从用户故事和验收标准中提取完整业务流程
  - *仅开发方案*：从接口命名规律推断基本 CRUD 流程
- **数据依赖场景**：识别接口间的数据传递关系，测试上游接口输出作为下游接口输入
  - *有需求文档*：从业务规则中识别数据关联和依赖关系
  - *仅开发方案*：从接口请求/响应参数中识别 ID 类字段的传递
- **状态流转场景**：识别业务状态机（如：订单状态、活动状态等），测试状态的正向流转和逆向流转
  - *有需求文档*：从验收标准中提取状态流转规则和非法状态拦截逻辑
  - *仅开发方案*：从接口响应中的 status/state 字段推断状态流转
- **异常场景**：业务规则约束、错误处理、边界条件
  - *有需求文档*：从 `acceptance_criteria` 中的 negative 类型场景、`business_rules` 中的约束条件提取
  - *仅开发方案*：基于接口参数的必填性和类型推断基本异常场景
- **用户角色场景**（仅当有需求文档时）：
  - 不同用户角色的权限差异测试
  - 从 `user_stories` 中的角色字段提取
- **并发与时序场景**：多接口并发调用的一致性，接口调用时序依赖测试，幂等性验证

**场景识别维度**（优先级从高到低）：

**当提供需求文档时，优先使用以下维度**：

1. **从验收标准反推场景**（最优先）：
   - 读取需求文档中的 `acceptance_criteria`
   - 每个验收标准对应至少一个场景用例
   - 识别 Given-When-Then 结构，转换为测试步骤
   - 关联到对应的接口调用

2. **从用户故事构建场景**：
   - 读取需求文档中的 `user_stories`
   - 识别用户角色（as）、操作（want）、目标（so that）
   - 构建端到端的用户操作流程
   - 识别不同角色的权限差异场景

3. **从业务规则提取边界和异常场景**：
   - 读取需求文档中的 `business_rules`
   - `data_rule` → 生成边界值测试场景
   - `process_rule` → 生成业务流程约束测试
   - `permission_rule` → 生成权限控制测试

4. **从标准化场景模板扩展**：
   - 读取需求文档中的 `scenarios`（如果有）
   - 直接映射到接口调用
   - 补充技术细节（请求参数、响应参数）

**当仅有开发方案时，使用技术推断维度**：

5. **接口命名规律识别**：
   - CRUD 操作：create/add/save → get/query/list → update/modify/edit → delete/remove
   - 状态流转：submit → review/audit → approve/reject → publish/close
   - 业务流程：register → verify → activate → use → expire/renew

6. **数据依赖关系识别**：
   - 查找返回 ID 类字段的接口（如：userId, orderId, activityId等）
   - 查找需要 ID 作为入参的接口
   - 建立数据流向关系链

7. **业务领域聚合**：
   - 同一业务领域的接口（如：活动相关、订单相关、用户相关）
   - 识别领域内的完整业务闭环

8. **状态机识别**：
   - 识别 status、state 类字段
   - 分析状态的可能流转路径
   - 识别状态流转的触发接口

> 📖 **详细指南**：参见 `references/03-scenario-identification.md` - 场景测试用例识别指南

## 工作流程

### 模式一：仅有开发方案文档（基础模式）

#### 步骤 1：文档接收与预处理
```
接收开发方案文档 → 文档格式识别 → 内容结构化解析 → 文档内容分析 → 检查文档质量
```

#### 步骤 2：接口信息提取
```
接口定义识别 → 接口路径校验（检测dmb/识别微服务）→ 参数信息提取 → 信息结构化整理 → 接口关系分析
```

#### 步骤 3：业务场景识别（技术推断）
```
分析业务流程（从接口命名推断）→ 识别接口依赖关系（从参数推断）→ 提取业务状态流转 → 识别数据传递链路 → 场景分类归纳
```

#### 步骤 4：测试用例设计
```
单接口用例设计 → 基础场景用例设计 → 接口风险评估 → 优先级排序
```

#### 步骤 5：输出与建议
```
生成测试左移文档（包含单接口用例 + 基础场景用例）
```

---

### 模式二：开发方案 + 需求文档（完整模式，推荐）

#### 步骤 1：需求文档分析（优先）
```
接收需求文档 → 识别文档格式（原始/标准化）→ 提取用户故事 → 提取验收标准 → 提取业务规则 → 提取测试场景
```

#### 步骤 2：开发方案文档分析
```
接收开发方案文档 → 文档格式识别 → 检查文档质量
```

#### 步骤 3：接口信息提取
```
接口定义识别 → 接口路径校验 → 参数信息提取 → 与需求文档中的功能关联
```

#### 步骤 4：业务场景映射与扩展
```
从验收标准生成场景 → 从用户故事生成端到端场景 → 从业务规则生成边界和异常场景 → 技术细节补充（接口调用） → 场景优先级排序
```

#### 步骤 5：测试用例设计
```
单接口用例设计（契约测试）→ 完整场景用例设计（业务流程测试）→ 接口风险评估 → 优先级排序
```

#### 步骤 6：输出与建议
```
生成完整的测试左移文档（包含单接口用例 + 全面的场景用例 + 需求覆盖度分析）
```

## 输出格式

生成的测试左移分析报告应包含以下部分（根据输入文档动态调整）：

### 1. 文档分析概览

#### 1.1 输入文档清单
- 需求文档：[文档名称]（如有）
- 开发方案文档：[文档名称]
- 是否使用标准化需求文档：是/否

#### 1.2 文档质量检查

**开发方案文档规范性检查**：
- 检查结果（✅ 符合规范 / ❌ 不符合规范）
- 检查项列表
- 改进建议（如不符合规范）

**需求文档完整性检查**（如有需求文档）：
- 是否包含用户故事：是/否
- 是否包含验收标准：是/否
- 是否包含业务规则：是/否
- 覆盖程度评估（高/中/低）

### 2. 接口路径校验结果
- 网关接口检测（如检测到包含 `dmb` 的接口，列出并提示用户替换）
- 微服务接口识别（列出所有识别的微服务接口及完整路径）

### 3. 接口信息汇总
为每个接口提供：
- 所属微服务
- 接口路径（完整格式：微服务域名 + 接口路径）
- 请求方法
- 功能描述
- 请求参数（JSON格式 + 参数说明表格）
- 响应参数（JSON格式 + 参数说明表格）

### 4. 测试左移用例推荐

#### 一、单接口测试用例
为每个接口提供：
- 功能测试（正常场景）
- 边界值测试
- 异常测试（必填参数缺失、参数格式错误、参数类型错误）
- 业务规则验证

每个测试用例包含：
- 用例描述
- 前置条件
- 测试步骤
- 请求参数（JSON格式）
- 预期结果（JSON格式）

#### 二、场景测试用例

**当提供需求文档时，增加以下内容**：
- **需求覆盖情况**：列出每个场景对应的需求文档中的验收标准（AC ID）或用户故事（US ID）
- **业务规则验证**：标注每个场景验证的业务规则（BR ID）

为每个识别的业务场景提供：
- 场景描述（完整的业务场景和业务目标）
  - *有需求文档*：引用用户故事描述
  - *仅开发方案*：基于接口功能总结
- 场景来源（新增）：
  - `需求文档-AC001`（验收标准）
  - `需求文档-US001`（用户故事）
  - `需求文档-BR001`（业务规则）
  - `接口推断`（从开发方案推断）
- 涉及接口（列出所有相关接口及功能说明）
- 接口调用关系图（文本格式）
- 测试用例（正向流程 + 至少1个异常流程）

每个测试用例包含：
- 用例描述
- 前置条件
  - *有需求文档*：从 Given 子句提取
  - *仅开发方案*：基于接口依赖推断
- 测试步骤（每步包含请求参数、预期响应）
- 验证点（使用 ✓ 标记）
  - *有需求文档*：从 Then 子句提取验证点
  - *仅开发方案*：基于响应参数推断
- **数据传递标记**：使用 `{{步骤N返回的字段名}}` 标记数据依赖

#### 三、场景用例优先级建议
以表格形式提供：
- 优先级（P0/P1/P2/P3）
- 场景类型
- 场景名称
- 理由
- 预计工时（可选）

## 最佳实践建议

### 文档质量要求
- 开发方案文档应包含完整的接口设计信息，或者 UDoc 链接
- 建议使用标准的 API 文档格式（如 OpenAPI/Swagger）
- 接口参数说明应详细完整

### 测试左移策略
- **早期介入**：在需求评审阶段就开始测试分析
- **持续跟进**：随开发进度更新测试用例
- **风险驱动**：重点关注高风险、核心业务接口
- **自动化优先**：推荐可自动化的测试用例设计
- **场景优先**：优先设计端到端的业务场景用例，覆盖完整业务流程
- **分层测试**：单接口用例保证接口稳定性，场景用例保证业务正确性

### 协作建议
- 与开发团队保持密切沟通
- 定期更新接口变更信息
- 建立接口测试基线
- 持续优化测试策略

### 场景用例设计原则
- **真实性**：场景应反映真实的业务使用情况
- **完整性**：覆盖业务流程的各个环节
- **可验证性**：每个步骤都有明确的验证点
- **可维护性**：场景描述清晰，易于理解和维护
- **数据关联**：明确标注接口间的数据传递关系（如：使用 `{{步骤1返回的ID}}`）

## 注意事项

### 场景识别要点
- 优先识别业务主流程，再识别异常分支
- 关注接口的数据依赖关系，不是所有接口都需要组合成场景
- 场景数量不宜过多，聚焦核心业务流程（通常 3-5 个场景即可）

### 用例设计要点
- 单接口用例聚焦接口自身的功能和契约
- 场景用例聚焦业务流程和数据一致性
- 每个场景用例应包含正向流程 + 至少1个异常流程

### 数据传递标记
- 使用 `{{步骤N返回的字段名}}` 标记数据依赖
- 便于测试执行时进行数据提取和传递

## 使用指南

### 基础用法

1. **仅提供开发方案文档**（基础模式）：
   ```
   请分析这个开发方案文档，生成测试左移分析报告
   文档路径：./docs/开发方案V1.0.docx
   ```

2. **同时提供需求文档和开发方案**（完整模式，推荐）：
   ```
   请结合需求文档和开发方案文档，生成详细的测试左移分析报告
   需求文档：./docs/需求文档V1.0.docx
   开发方案：./docs/开发方案V1.0.docx
   ```

3. **使用标准化需求文档**（最优模式）：
   ```
   请基于标准化需求文档和开发方案文档，生成测试左移分析报告
   标准化需求文档：./docs/normalized_requirement.yaml
   开发方案：./docs/开发方案V1.0.docx
   ```

### 高级选项

1. **指定分析重点**：可以指定关注的特定模块或接口
2. **选择输出详细度**：可以要求详细分析或概要分析
3. **定制化需求**：可以根据项目特点提出特殊要求
4. **场景优先**：如果文档包含多个相关接口，优先生成场景测试用例

### 最佳实践

1. **优先使用完整模式**：同时提供需求文档和开发方案，生成更全面的场景用例
2. **使用标准化需求**：如果条件允许，先用 `req-parser` 生成标准化需求文档
3. **补充 UDoc 链接**：如果开发方案中提供 UDoc 链接，可以获取更准确的接口信息
4. **迭代优化**：首次分析后，根据缺失项清单补充文档，重新分析
