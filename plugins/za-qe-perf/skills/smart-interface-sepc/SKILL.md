# 测试点分析任务

## 📋 分析范围

$ARGUMENTS

**如果未指定文件/目录**：自动分析git最近修改的代码（git diff）

**指定方式示例**：
- 单个文件: `/test-point-analysis src/main/java/group/za/bank/trade/controller/TradeController.java`
- 多个文件: `/test-point-analysis File1.java File2.java`
- 整个目录: `/test-point-analysis src/main/java/group/za/bank/trade/`
- Git修改: `/test-point-analysis` (不带参数)
- 保存到指定目录: `/test-point-analysis -o ./test-results` 或 `/test-point-analysis --output ./test-results`
- **分支对比**: `/test-point-analysis --branch main feature/my-branch` 或 `/test-point-analysis -b main feature/my-branch`
- **当前改动 vs 分支**: `/test-point-analysis --branch main` 或 `/test-point-analysis -b main`（只传一个分支，对比当前工作区与该分支的差异）

## 🔍 分析步骤

### Step 0: 解析参数

#### 0.1 解析参数
从 `$ARGUMENTS` 中提取以下参数：
- `-o <路径>` 或 `--output <路径>`: 文件保存目录路径（可选，不指定则使用默认目录）
- `--branch <分支A> <分支B>` 或 `-b <分支A> <分支B>`: 分支对比模式
- 其余内容为分析范围（文件/目录）

**分支对比模式识别**：
- **传入两个分支**：`--branch <base分支> <compare分支>`，对比两个历史分支的差异
- **传入一个分支**：`--branch <base分支>`，对比当前工作区（含未提交改动）与该分支的差异
- 示例：
  - `--branch main feature/order-refactor` → 分析 feature/order-refactor 相对于 main 的所有改动
  - `--branch main` → 分析当前本地文件（工作区+暂存区）相对于 main 分支的所有改动

#### 0.2 准备文件输出
1. 确定输出目录路径：
   - 如果指定了 `-o`/`--output`，使用该路径
   - 否则使用默认目录 `./test-point-analysis-results`
2. **强制执行以下 shell 命令获取当前时间戳**，禁止手动填写或猜测时间：
   ```bash
   date +"%Y%m%d-%H%M%S"
   ```
   将命令输出结果作为时间戳，生成文件名格式：`test-point-analysis-{时间戳}.md`
3. 记录完整输出路径备用

### Step 1: 读取代码变更
获取需要分析的代码变更

**分支对比模式下的代码获取**：
- 如果参数中包含 `--branch` 或 `-b`，根据后续分支参数数量选择对应命令：

  **情况A：传入两个分支（`--branch <base> <compare>`）** → 对比两个历史分支：
  ```bash
  git diff --name-only <base分支> <compare分支>
  git diff <base分支> <compare分支>
  git show <compare分支>:<文件路径>
  ```

  **情况B：只传入一个分支（`--branch <base>`）** → 对比当前工作区与目标分支：
  ```bash
  git diff --name-only <base分支>
  git diff <base分支>
  # 直接读取本地当前文件内容
  ```

- 优先分析变更文件的**完整内容**，同时参考 diff 了解具体改动点

### Step 2: 激活相关Skills
根据代码类型自动激活相关Skills：
- **invest-framework-overview**: 框架整体规范
- **invest-controller-spec**: Controller层规范
- **invest-service-spec**: Service层规范
- **invest-dao-spec**: DAO层规范

- **invest-exception-spec**: 异常处理规范
- **invest-database-guide**: 数据库ORM规范
- **invest-feign-guide**: Feign调用规范

### Step 3: 全面识别所有变更接口点（关键步骤）

**此步骤必须穷举所有变更点，不能遗漏任何变更细节。**

#### 3.1 逐行分析diff，提取代码变更点
对每个变更文件的diff进行逐行分析，识别以下变更类型：

| 变更类型 | 识别标志 | 分析要点 |
|---------|---------|---------|
| **新增接口** | `@PostMapping`/`@GetMapping`等新增方法 | URL、入参、出参、校验规则 |
| **修改接口** | 已有方法体内代码变更 | 新增/删除/修改的逻辑分支 |
| **删除接口** | 方法/接口被删除 | 影响范围评估 |
| **新增Service方法** | Service/Manager新增方法 | 调用方、被调用方、SQL操作 |
| **修改Service方法** | 方法体内逻辑变更 | 业务规则变化、异常处理变化 |
| **新增Mapper方法** | 新增SQL操作 | 表名、字段、条件、索引 |
| **修改Mapper方法** | SQL语句变更 | 字段增减、条件变化、join变化 |
| **新增Feign接口** | FeignClient新增方法 | 远程服务、URL、入参、出参 |
| **修改Feign接口** | Feign方法变更 | 远程契约变化 |
| **新增/修改Job** | @XxlJob/@Scheduled变更 | 触发条件、执行逻辑 |
| **新增/修改DTO** | 字段增减、类型变化、校验规则变化 | 影响所有使用该DTO的接口 |
| **新增/修改Entity** | 字段增减、注解变化 | 影响数据库操作 |
| **新增/修改配置** | yml/xml/properties变更 | 运行时行为变化 |
| **新增/修改异常** | try-catch/throw变更 | 错误处理路径变化 |
| **新增/修改事务** | TransactionUtils变更 | 事务边界变化 |

#### 3.2 构建调用链，识别受影响的接口
对每个变更点，沿调用链向上追溯，找出所有受影响的对外接口：

1. **Controller → Service → DAO 方向**：
   - 变更在Service层 → 找到调用该Service的所有Controller方法
   - 变更在DAO/Mapper层 → 找到调用该Mapper的所有Service方法 → 再找到对应Controller
   - 变更在DTO/Entity → 找到所有使用该DTO/Entity的接口

2. **Feign调用方向**：
   - 变更在Feign接口 → 找到调用该Feign的Service → 再找到对应Controller
   - 变更在Service中被Feign调用的方法 → 影响下游消费者

3. **MQ消息方向**：
   - 变更在消息生产逻辑 → 影响消息消费者
   - 变更在消息消费逻辑 → 影响消息生产者和其他消费者

4. **Job/定时任务方向**：
   - 变更在Job → 评估对依赖数据/下游系统的影响

#### 3.3 输出变更接口清单
必须列出**所有**识别到的变更接口，包括：
- 直接变更的接口（代码本身有变更）
- 间接影响的接口（调用链上游的对外接口）
- 每条变更接口需注明：
  - 变更类型（新增/修改/删除/间接影响）
  - 变更的具体内容（字段增减、逻辑变化、参数变化等）
  - 变更风险等级（🔴高/🟡中/🟢低）

### Step 4: 生成测试点分析报告
**基于Step 3中识别到的所有变更接口点，逐一对应生成测试点。**

**约束**：
- 每个直接变更的接口必须有对应的测试点
- 每个间接影响的接口必须有对应的回归测试点
- 每个代码变更逻辑分支（新增if-else、循环变更、异常处理变更）必须有对应的测试点
- 不得遗漏Step 3中识别到的任何变更点

## 1️⃣ 接口级功能测试点 🧪

**⚠️ 必须为Step 3中识别到的每个变更接口生成对应的功能测试点，不得遗漏。**

### 新增接口测试
- 为每个新增的Controller接口生成正向和反向测试用例
- 为每个新增的Service方法生成单元测试点
- 为每个新增的Job任务生成执行测试点

### 修改接口测试
- **针对每个被修改的接口**，对比diff识别具体变更点：
  - 入参变更：新增字段、删除字段、类型变化、校验规则变化
  - 出参变更：返回字段增减、数据结构变化
  - 业务逻辑变更：新增/删除/修改的if-else分支、循环、计算逻辑
  - 异常处理变更：新增异常抛出、异常类型变化
- 为每个变更点生成对应的测试用例
- 生成变更前的回归测试点（确保未变更的行为不受影响）

### 删除接口测试
- 识别被删除的功能点和接口
- 生成确认删除无副作用的测试点
- 检查是否有其他模块/前端/外部系统依赖被删除的功能

### Feign调用变更测试
- 识别新增/修改的Feign Client方法

### MQ消息变更测试

### Job/定时任务变更测试
- 识别新增/修改的Job任务
- 生成触发条件、执行逻辑、幂等性测试点

### 业务场景覆盖

- **基金交易场景**: 申购、赎回、转换、分红、撤单等

## 📋 分析原则

1. **基于变更**: 测试点紧扣代码变更，不凭空捏造
2. **风险导向**: 高风险变更分配更多测试点和更高优先级
3. **覆盖全面**: 正向、反向、边界、异常都要覆盖
4. **可执行**: 测试点要具体可执行，不是泛泛而谈
5. **优先级分明**: P0（必须测）、P1（重要）、P2（一般）
6. **回归意识**: 关注变更对已有功能的影响

## 🎯 使用示例

### 示例1: 分析git diff（默认行为）
```bash
/test-point-analysis
```
**行为**: 自动分析git diff的修改，保存到 `./test-point-analysis-results/test-point-analysis-YYYYMMDD-HHmmss.md`

### 示例2: 分析指定文件
```bash
/test-point-analysis src/main/java/group/za/bank/trade/controller/TradeController.java
```
**输出**: 保存到默认目录

### 示例3: 保存到指定目录
```bash
/test-point-analysis -o ./reports src/main/java/group/za/bank/trade/
/test-point-analysis --output /tmp/test-points File1.java File2.java
```
**输出**: 保存到指定路径

### 示例4: 对比两个分支进行分析
```bash
# 分析 feature 分支相对于 main 的所有改动（两分支对比）
/test-point-analysis --branch main feature/order-refactor

# 分析当前本地文件相对于 main 分支的改动（单分支模式，含未提交改动）
/test-point-analysis --branch main
/test-point-analysis -b main

# 结合输出参数
/test-point-analysis --branch main -o ./test-reports
/test-point-analysis -b main --output ./test-reports

# 短参数形式（两分支）
/test-point-analysis -b main feature/order-refactor
```
**两分支模式行为**: 执行 `git diff main feature/order-refactor`，获取变更文件列表，读取文件内容进行分析

**单分支模式行为**: 执行 `git diff main`，获取当前工作区相对于 main 的变更文件，读取本地当前文件内容进行分析（包含所有未提交改动）

**重要提示**:
- ✅ 结果始终保存到文件，窗口只显示摘要
- ✅ 文件名含时间戳，不会覆盖历史报告
- ✅ 支持相对路径和绝对路径，目录不存在时自动创建
- ✅ `--branch`/`-b <base> <compare>` 自动通过 git diff 获取分支差异，无需手动指定文件
- ✅ `-o`/`--output` 用于指定保存目录
