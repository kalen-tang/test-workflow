# 需求文档与开发方案文档结合分析指南

## 概述

本文档说明如何同时使用需求文档和开发方案文档，生成更全面、更贴近业务的测试左移场景用例。

---

## 为什么需要结合两种文档

### 仅使用开发方案文档的局限性

开发方案文档通常侧重于**技术实现**：
- ✅ 提供详细的接口定义（路径、参数、响应）
- ✅ 说明技术架构和实现方案
- ⚠️ 业务场景描述不够详细
- ⚠️ 缺少完整的用户故事和验收标准
- ⚠️ 异常场景和边界条件描述不全
- ⚠️ 可能缺少业务规则的完整说明

**结果**：
- 单接口测试用例：**质量高**（基于详细的接口定义）
- 场景测试用例：**不够全面**（仅能从接口命名和参数依赖推断）

### 结合需求文档的优势

需求文档侧重于**业务需求**：
- ✅ 完整的用户故事（用户角色、操作、目标）
- ✅ 明确的验收标准（Given-When-Then）
- ✅ 详细的业务规则和约束条件
- ✅ 异常场景和边界条件的完整描述
- ⚠️ 缺少技术实现细节（接口路径、参数格式）

**结果**：
- 单接口测试用例：**需要补充技术细节**
- 场景测试用例：**业务场景完整**（基于真实的用户故事和验收标准）

### 最佳实践：两者结合

| 维度 | 开发方案文档 | 需求文档 | 结合后效果 |
|------|-------------|---------|-----------|
| **接口技术规格** | ✅ 完整 | ❌ 缺失 | ✅ 完整 |
| **业务场景描述** | ⚠️ 不详细 | ✅ 完整 | ✅ 完整 |
| **验收标准** | ⚠️ 不明确 | ✅ 明确 | ✅ 明确 |
| **异常场景** | ⚠️ 技术异常 | ✅ 业务异常 | ✅ 全面覆盖 |
| **测试用例质量** | ⚠️ 中等 | ⚠️ 中等 | ✅ 优秀 |

## 工作流程

### 第一步：需求文档分析

#### 1.1 识别文档格式

**原始需求文档**（Word/PDF/Markdown）：
- 需要手工提取和解析
- 可能结构不规范，需要智能识别

**标准化需求文档**（YAML 格式）：
- 已经过 `requirement-normalizer` 处理
- 结构化数据，直接读取
- 推荐使用这种格式

#### 1.2 提取关键信息

从需求文档中提取以下信息：

**A. 功能列表（Features）**
```yaml
features:
  - id: F001
    name: "用户登录"
    description: "用户可通过手机号和密码登录系统"
    priority: high
    test_focus: true
    risk_level: medium
```

**用途**：
- 了解业务功能全貌
- 识别测试重点（`test_focus: true`）
- 评估风险级别（`risk_level`）

---

**B. 用户故事（User Stories）**
```yaml
user_stories:
  - id: US001
    as: "普通用户"
    want: "通过手机号快速登录"
    so_that: "无需记住复杂的用户名"
    relates_to: F001
```

**用途**：
- 构建端到端的用户操作流程
- 识别不同用户角色的差异场景
- 生成场景用例的业务描述

**转换为场景用例**：
```
场景：普通用户快速登录
场景描述：作为普通用户，我希望通过手机号快速登录，以便无需记住复杂的用户名
```

---

**C. 验收标准（Acceptance Criteria）**
```yaml
acceptance_criteria:
  - id: AC001
    description: "用户输入正确的手机号和密码，点击登录按钮，跳转到首页"
    relates_to: F001
  - id: AC002
    description: "用户输入错误的密码，显示错误提示'密码错误'"
    relates_to: F001
```

**用途**：
- 直接转换为测试场景
- 识别 Given-When-Then 结构
- 确定验证点

**转换为场景用例**：
```
场景用例1：正常登录流程（AC001）
前置条件（Given）：用户已注册，手机号为13800138000，密码为Test1234
操作步骤（When）：用户输入手机号13800138000，密码Test1234，点击登录
预期结果（Then）：跳转到首页

场景用例2：错误密码登录（AC002）
前置条件（Given）：用户已注册，手机号为13800138000，正确密码为Test1234
操作步骤（When）：用户输入手机号13800138000，密码WrongPass，点击登录
预期结果（Then）：停留在登录页，显示错误提示"密码错误"
```

---

**D. 业务规则（Business Rules）**
```yaml
business_rules:
  - id: BR001
    rule: "手机号必须为11位数字，且以1开头"
    type: data_rule
    relates_to: [F001, F002]
  - id: BR003
    rule: "用户连续输错密码3次，账号锁定30分钟"
    type: process_rule
    relates_to: [F001]
```

**用途**：
- 生成边界值测试用例
- 生成异常场景用例
- 验证业务约束

**转换为场景用例**：
```
场景用例3：手机号格式校验（BR001）
前置条件：用户在登录页
操作步骤：用户输入手机号12345678（不足11位），点击登录
预期结果：显示提示"手机号格式错误"

场景用例4：连续输错密码锁定（BR003）
前置条件：用户已连续输错密码2次
操作步骤：用户第3次输入错误密码
预期结果：显示"账号已锁定30分钟"，登录按钮禁用
```

---

**E. 测试场景模板（Scenarios）**
```yaml
scenarios:
  - id: S001
    name: "正常登录-正确账号密码"
    type: positive
    relates_to: AC001
    given: "用户已注册，手机号为13800138000，密码为Test1234"
    when: "用户输入手机号13800138000，密码Test1234，点击登录"
    then: "跳转到首页，显示用户昵称"
```

**用途**：
- 直接使用，无需额外转换
- 补充接口技术细节（从开发方案文档获取）

### 第三步：场景映射与扩展

#### 3.1 从验收标准生成场景

**验收标准**：
```
AC001: 用户输入正确的手机号和密码，点击登录按钮，跳转到首页
```

**映射到接口**：
- 步骤1：调用 `POST /user/login`
- 请求参数：`{ "phone": "13800138000", "password": "Test1234" }`
- 预期响应：`{ "code": 200, "data": { "userId": "U123", "token": "..." } }`
- 验证点：✓ 返回 code=200，✓ 返回 userId 和 token

**生成场景用例**：
```markdown
### 场景1：用户正常登录（AC001）

**场景来源**：需求文档-AC001

**场景描述**：
用户输入正确的手机号和密码，成功登录系统，跳转到首页。

**涉及接口**：
1. POST /user/login - 用户登录

**测试用例1：正向流程-正确账号密码**

**前置条件**：
- 用户已注册，手机号为13800138000，密码为Test1234

**测试步骤**：

步骤1：调用登录接口
- 请求参数：
```json
{
  "phone": "13800138000",
  "password": "Test1234"
}
```
- 预期响应：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "userId": "U123456",
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "nickname": "张三"
  }
}
```

**验证点**：
- ✓ 返回 code=200
- ✓ 返回有效的 userId
- ✓ 返回有效的 token
- ✓ 返回用户昵称
```

#### 3.2 从用户故事生成端到端场景

**用户故事**：
```
US001: 作为普通用户，我希望通过邮箱验证加入企业团队，以便获得企业专属权益
```

**关联接口**（从开发方案文档识别）：
1. `POST /activity/getVerifyCode` - 获取验证码
2. `POST /activity/verify` - 企业认证
3. `GET /activity/entryPage` - 查询活动状态
4. `GET /activity/homePage` - 查询权益列表

**生成端到端场景**：
```markdown

**场景来源**：需求文档-US001

**场景描述**：
作为普通用户，我希望通过邮箱验证加入企业团队，以便获得企业专属权益。

**涉及接口**：
1. POST /activity/getVerifyCode - 获取验证码
2. POST /activity/verify - 企业认证
3. GET /activity/entryPage - 查询活动状态
4. GET /activity/homePage - 查询权益列表

**接口调用关系图**：
```
getVerifyCode（获取验证码） → verify（企业认证） → entryPage（查询状态） → homePage（查看权益）
      ↓                          ↓                    ↓                     ↓
  返回tokenId                使用tokenId          验证joinFlag=true    返回权益列表
```

**测试用例1：正向流程-新用户成功加入**

**前置条件**：
- 用户未参与活动
- 邀请码有效
- 用户邮箱为 newuser@company.com

**测试步骤**：

步骤1：调用 getVerifyCode 获取验证码
- 请求参数：
```json
{
  "revIdType": "EMAIL",
  "revId": "newuser@company.com",
  "redeemCode": "ABC123"
}
```
- 预期响应：
```json
{
  "code": "000000",
  "value": { "tokenId": "TOKEN_001" },
  "success": true
}
```

步骤2：使用步骤1返回的tokenId，调用 verify 进行认证
- 请求参数：
```json
{
  "customerNo": "C_NEW_001",
  "tokenId": "{{步骤1返回的tokenId}}",
  "revIdType": "EMAIL",
  "revId": "newuser@company.com",
  "verifyCode": "123456",
  "redeemCode": "ABC123"
}
```
- 预期响应：
```json
{
  "code": "000000",
  "value": {
    "currentLevel": 1,
    "levelChange": true,
    "advCodeUseSuccessFlag": true
  },
  "success": true
}
```

步骤3：调用 entryPage 验证参与状态
- 请求参数：
```json
{
  "customerNo": "C_NEW_001"
}
```
- 预期响应：
```json
{
  "code": "000000",
  "value": {
    "joinFlag": true,
    "currentLevel": 1
  },
  "success": true
}
```

步骤4：调用 homePage 查询权益列表
- 请求参数：
```json
{
  "customerNo": "C_NEW_001"
}
```
- 预期响应：包含权益列表

**验证点**：
- ✓ 每个步骤返回的状态码正确
- ✓ tokenId在步骤1和步骤2之间正确传递

## 场景优先级评估

结合需求文档和开发方案，可以更准确地评估场景优先级：

| 优先级 | 评估维度 | 需求文档提供 | 开发方案提供 | 综合判断 |
|-------|---------|------------|------------|---------|
| **P0** | 业务重要性 | `priority: high` | 核心接口 | 必须测试 |
| **P0** | 测试重点 | `test_focus: true` | - | 必须测试 |
| **P1** | 风险级别 | `risk_level: high` | 涉及资金/安全 | 重点测试 |
| **P1** | 异常场景 | `type: negative` | 错误码完整 | 重点测试 |
| **P2** | 边界场景 | `type: boundary` | 参数约束 | 常规测试 |
| **P3** | 扩展场景 | - | 次要接口 | 可选测试 |

## 最佳实践总结

### 1. 文档准备

✅ **推荐做法**：
- 先用 `requirement-normalizer` 生成标准化需求文档
- 开发方案文档包含完整的接口定义或 UDoc 链接
- 两个文档都准备好后，一起提供给 `shift-left-analyzer`

❌ **不推荐做法**：
- 需求文档和开发方案内容不一致
- 开发方案缺少接口定义
- 需求文档过于简略，缺少验收标准

### 2. 场景用例生成策略

- **从验收标准出发**：每个验收标准对应至少一个场景用例
- **补充异常场景**：从业务规则中提取边界和异常条件
- **构建端到端流程**：从用户故事中提取完整操作流程
- **标注来源**：每个场景标注来源（需求文档 AC001 / 接口推断）

### 3. 输出质量保证

- **需求追溯**：每个场景用例关联到需求文档的 ID（AC001、US001、BR001）
- **接口映射**：每个场景明确列出涉及的接口
- **验证点明确**：从需求文档的 Then 子句提取验证点
- **数据传递清晰**：使用 `{{步骤N返回的字段名}}` 标记数据依赖

### 4. 常见问题

**Q1：需求文档和开发方案描述不一致怎么办？**
- 在报告中标注不一致项
- 提示用户确认以哪个为准
- 建议同步更新文档

**Q2：需求文档没有标准化，如何处理？**
- 优先识别 Given-When-Then 结构
- 手工提取用户故事和验收标准
- 建议用户使用 `requirement-normalizer` 标准化

**Q3：开发方案缺少某些接口的定义怎么办？**
- 标注缺失项
- 基于需求文档推断接口功能
- 提示用户补充开发方案

## 示例对比

### 仅使用开发方案文档

```markdown
### 场景1：创建活动到发布完整流程

**场景来源**：接口推断

**场景描述**：
测试从创建活动、审核活动到发布活动的完整流程（基于接口命名推断）

**涉及接口**：
1. POST /activity/create - 创建活动
2. POST /activity/audit - 审核活动
3. POST /activity/publish - 发布活动

**测试用例**：正向流程（略）
```

**特点**：
- ✅ 接口技术规格准确
- ⚠️ 业务场景描述较简略
- ⚠️ 缺少验收标准和验证点
- ⚠️ 异常场景不全面

---

### 结合需求文档和开发方案

```markdown
### 场景1：运营人员创建并发布营销活动（US002 + AC005 + AC006）

**场景来源**：
- 需求文档-US002（用户故事）
- 需求文档-AC005（验收标准-创建成功）
- 需求文档-AC006（验收标准-发布成功）
- 需求文档-BR008（业务规则-活动审核）

**场景描述**：
作为运营人员，我希望创建营销活动并提交审核，审核通过后发布活动，以便用户可以参与活动。

**涉及接口**：
1. POST /activity/create - 创建活动
2. POST /activity/audit - 审核活动
3. POST /activity/publish - 发布活动
4. GET /activity/get - 查询活动详情（验证状态）

**业务规则验证**：
- BR008：活动必须经过审核才能发布

**测试用例1：正向流程-创建到发布**

**前置条件**（来自AC005）：
- 运营人员已登录
- 具有创建和发布活动的权限
- 活动时间设置合理（开始时间 < 结束时间）

**测试步骤**：

步骤1：创建活动
步骤2：审核活动（审核通过）
步骤3：发布活动
步骤4：查询活动详情，验证状态为"已发布"

**验证点**（来自AC005、AC006）：
- ✓ 创建成功后返回活动ID
- ✓ 活动状态从"草稿"变为"已审核"
- ✓ 审核通过后可以发布
- ✓ 发布后状态变为"已发布"
