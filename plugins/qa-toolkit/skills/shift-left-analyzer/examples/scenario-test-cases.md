## 示例 1：CRUD 完整流程场景

### 场景：用户管理完整流程

**场景描述**：
测试用户创建、查询、更新、删除的完整 CRUD 流程，验证数据的增删改查功能正常，状态流转正确。

**涉及接口**：
1. 接口A：创建用户（`POST /user/create`）- 返回 userId
2. 接口B：查询用户（`GET /user/get`）- 使用 userId
3. 接口C：更新用户（`POST /user/update`）- 使用 userId
4. 接口D：删除用户（`DELETE /user/delete`）- 使用 userId

**接口调用关系图**：
```
接口A（创建） → 接口B（查询验证） → 接口C（更新） → 接口B（查询验证） → 接口D（删除）
    ↓                ↑                  ↓                ↑
  返回ID          使用ID              使用ID           使用ID
```

---

#### 测试用例1：正向流程-CRUD完整操作

**测试步骤**：

##### 步骤1：创建用户

- **请求参数**：
```json
{
  "userName": "张三",
  "email": "zhangsan@example.com",
  "phone": "13800138000"
}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "userId": "U001",
    "userName": "张三",
    "status": "ACTIVE"
  }
}
```

##### 步骤2：查询用户，验证创建结果

- **请求参数**：
```
GET /user/get?userId={{步骤1返回的userId}}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "userId": "U001",
    "userName": "张三",
    "email": "zhangsan@example.com",
    "phone": "13800138000",
    "status": "ACTIVE"
  }
}
```

##### 步骤3：更新用户信息

- **请求参数**：
```json
{
  "userId": "{{步骤1返回的userId}}",
  "userName": "张三丰",
  "email": "zhangsanfeng@example.com"
}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "userId": "U001",
    "userName": "张三丰",
    "email": "zhangsanfeng@example.com"
  }
}
```

##### 步骤4：再次查询用户，验证更新结果

- **请求参数**：
```
GET /user/get?userId={{步骤1返回的userId}}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "userId": "U001",
    "userName": "张三丰",  // 已更新
    "email": "zhangsanfeng@example.com",  // 已更新
    "phone": "13800138000",  // 未更新，保持原值
    "status": "ACTIVE"
  }
}
```

##### 步骤5：删除用户

- **请求参数**：
```
DELETE /user/delete?userId={{步骤1返回的userId}}
```

- **预期响应**：
```json
{
  "code": 200,
  "message": "用户删除成功"
}
```

##### 步骤6：再次查询用户，验证删除结果

- **请求参数**：
```
GET /user/get?userId={{步骤1返回的userId}}
```

- **预期响应**：
```json
{
  "code": 404,
  "message": "用户不存在"
}
```

**验证点**：
- ✓ 创建成功，返回 userId
- ✓ 创建后立即可查询，数据一致
- ✓ 更新成功，只更新指定字段
- ✓ 更新后查询验证，数据已更新
- ✓ 删除成功
- ✓ 删除后查询不到数据

## 示例 2：状态流转场景

### 场景：订单状态流转完整流程

**场景描述**：
测试订单从创建到完成的状态流转，验证订单状态的正向流转和非法流转拦截。

**涉及接口**：
1. 接口A：创建订单（`POST /order/create`）- 返回 orderId
2. 接口B：支付订单（`POST /order/pay`）- 使用 orderId
3. 接口C：发货（`POST /order/ship`）- 使用 orderId
4. 接口D：确认收货（`POST /order/confirm`）- 使用 orderId
5. 接口E：查询订单（`GET /order/get`）- 验证状态

**状态流转图**：
```
PENDING（待支付） → PAID（已支付） → SHIPPED（已发货） → COMPLETED（已完成）
```

**接口调用关系图**：
```
接口A（创建） → 接口B（支付） → 接口C（发货） → 接口D（确认收货） → 接口E（查询验证）
  ↓ PENDING    ↓ PAID        ↓ SHIPPED    ↓ COMPLETED
  返回ID       使用ID         使用ID        使用ID
```

---

#### 测试用例1：正向流程-订单状态正常流转

**测试步骤**：

##### 步骤1：创建订单

- **请求参数**：
```json
{
  "productId": "P001",
  "quantity": 2,
  "price": 100
}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "orderId": "O001",

    "totalAmount": 200

}
```

##### 步骤2：支付订单

- **请求参数**：
```json
{
  "orderId": "{{步骤1返回的orderId}}",

}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "orderId": "O001",
    "status": "PAID"
  }
}
```

##### 步骤3：发货

- **请求参数**：
```json
{
  "orderId": "{{步骤1返回的orderId}}",
  "trackingNumber": "SF123456"
}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "orderId": "O001",
    "status": "SHIPPED"
  }
}
```

##### 步骤4：确认收货

- **请求参数**：
```json
{
  "orderId": "{{步骤1返回的orderId}}"
}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "orderId": "O001",
    "status": "COMPLETED"
  }
}
```

##### 步骤5：查询订单，验证最终状态

- **请求参数**：
```
GET /order/get?orderId={{步骤1返回的orderId}}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "orderId": "O001",
    "status": "COMPLETED",
    "productId": "P001",
    "quantity": 2,
    "totalAmount": 200
  }
}
```

**验证点**：
- ✓ 订单状态正确流转：PENDING → PAID → SHIPPED → COMPLETED
- ✓ 每个步骤返回正确的状态
- ✓ orderId 在整个流程中保持一致

## 示例 3：数据依赖场景

### 场景：积分兑换流程

**场景描述**：
测试查询积分、兑换商品、扣减积分的数据依赖流程，验证数据在流程中的一致性。

**涉及接口**：
1. 接口A：查询用户积分（`GET /points/query`）- 返回当前积分
2. 接口B：兑换商品（`POST /points/exchange`）- 使用积分
3. 接口C：再次查询积分（`GET /points/query`）- 验证扣减结果

**接口调用关系图**：
```
接口A（查询积分） → 接口B（兑换商品，扣减积分） → 接口C（验证积分扣减）
    ↓                      ↓                          ↑
  返回积分              使用积分                   验证积分
```

---

#### 测试用例1：正向流程-积分兑换成功

**测试步骤**：

##### 步骤1：查询用户当前积分

- **请求参数**：
```
GET /points/query?userId=U001
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "userId": "U001",
    "points": 1000
  }
}
```

##### 步骤2：兑换商品

- **请求参数**：
```json
{
  "userId": "U001",
  "productId": "P001",
  "requiredPoints": 500
}
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "orderId": "O001",
    "usedPoints": 500,
    "remainingPoints": 500
  }
}
```

##### 步骤3：再次查询积分，验证扣减

- **请求参数**：
```
GET /points/query?userId=U001
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "userId": "U001",
    "points": 500  // 1000 - 500 = 500
  }
}
```

**验证点**：
- ✓ 兑换前积分查询正确
- ✓ 兑换成功返回正确的扣减积分和剩余积分
- ✓ 兑换后积分正确扣减
- ✓ 数据一致性：步骤2的 remainingPoints = 步骤3的 points

---

#### 测试用例2：异常流程-积分不足

**测试步骤**：

##### 步骤1：查询积分（当前积分 500）

##### 步骤2：尝试兑换需要 1000 积分的商品

- **请求参数**：
```json
{
  "userId": "U001",
  "productId": "P002",
  "requiredPoints": 1000
}
```

- **预期响应**：
```json
{
  "code": 400,
  "message": "积分不足：当前积分 500，需要 1000"
}
```

##### 步骤3：查询积分，验证未扣减

- **请求参数**：
```
GET /points/query?userId=U001
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "userId": "U001",
    "points": 500  // 积分未变
  }
}
```

**验证点**：
- ✓ 积分不足时兑换失败
- ✓ 返回清晰的错误提示（包含当前积分和所需积分）
- ✓ 兑换失败后积分未扣减

## 示例 4：并发场景

### 场景：库存扣减并发测试

**场景描述**：
测试多用户并发购买商品时的库存扣减逻辑，验证库存不会超卖。

**涉及接口**：
1. 接口A：查询库存（`GET /inventory/query`）
2. 接口B：创建订单（`POST /order/create`）- 扣减库存

---

#### 测试用例1：并发购买-库存正确扣减

**测试步骤**：

##### 步骤1：查询商品库存

- **请求参数**：
```
GET /inventory/query?productId=P001
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "productId": "P001",
    "stock": 10
  }
}
```

##### 步骤2：并发创建 10 个订单

- **并发请求**（同时发起 10 个请求）：
```json
{
  "productId": "P001",
  "quantity": 1
}
```

- **预期结果**：
  - 10 个订单全部创建成功
  - 每个订单扣减 1 个库存

##### 步骤3：查询库存，验证扣减结果

- **请求参数**：
```
GET /inventory/query?productId=P001
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "productId": "P001",
    "stock": 0  // 10 - 10 = 0
  }
}
```

**验证点**：
- ✓ 并发请求全部成功
- ✓ 库存正确扣减（10 - 10 = 0）
- ✓ 无超卖现象

---

#### 测试用例2：并发购买-防止超卖

**测试步骤**：

##### 步骤1：查询商品库存（当前库存 5）

##### 步骤2：并发创建 10 个订单

- **并发请求**（同时发起 10 个请求）：
```json
{
  "productId": "P001",
  "quantity": 1
}
```

- **预期结果**：
  - 5 个订单创建成功
  - 5 个订单创建失败（库存不足）

##### 步骤3：查询库存，验证扣减结果

- **请求参数**：
```
GET /inventory/query?productId=P001
```

- **预期响应**：
```json
{
  "code": 200,
  "data": {
    "productId": "P001",
    "stock": 0  // 5 - 5 = 0（无超卖）
  }
}
```

**验证点**：
- ✓ 部分请求成功（5个）
- ✓ 部分请求失败并返回库存不足提示（5个）
- ✓ 库存正确扣减，无超卖（5 - 5 = 0）
- ✓ 并发情况下数据一致性保持

## 场景用例优先级评估

根据以上示例场景，评估优先级：

| 优先级 | 场景类型 | 场景名称 | 理由 |
|-------|---------|---------|------|
| **P0** | 核心业务流程 | CRUD 完整流程 | 基础功能，影响所有业务 |
| **P0** | 状态流转 | 订单状态流转完整流程 | 核心业务流程，涉及支付、发货等关键环节 |
| **P1** | 数据依赖 | 积分兑换流程 | 涉及资金类数据，必须保证一致性 |
| **P1** | 状态流转 | 非法状态流转拦截 | 保障业务规则正确性 |
| **P2** | 并发场景 | 库存扣减并发测试 | 高并发场景，防止超卖 |

## 数据传递标记规范

在场景用例中，使用统一的标记方式标注数据依赖：

```markdown
{{步骤N返回的字段名}}
```

**示例**：
```json
{
  "orderId": "{{步骤1返回的orderId}}",
  "userId": "{{步骤2返回的userId}}",
  "activityId": "{{步骤1返回的activityId}}"
}
```

**好处**：
- 清晰标识接口间的数据依赖关系
- 便于测试执行时进行数据提取和传递
- 易于维护和理解
