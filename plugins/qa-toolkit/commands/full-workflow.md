## 📊 完整模式四阶段流程

```
第一阶段：原始产出物规范化 🚧
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📄 原始需求文档 → 需求文档规范 Skill → 标准化需求文档
📄 原始设计文档 → 设计文档规范 Skill → 标准化设计文档
📄 原始历史案例 → CML MCP → 规范化历史案例
💻 开发代码 → Code Diff MCP → 代码变更分析报告
        ↓
📦 规范化产出物
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

第二阶段：需求实现检查 ✅
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📦 规范化产出物（需求、设计、代码差异）
        ↓
✅ requirement-validator
        ↓
📊 需求实现检查报告
   - 文档质量评分 (A/B/C/D)
   - 需求对齐度分析
   - 测试重点和风险标注
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

第三阶段：手工案例生成 🚧
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📦 规范化产出物（需求、设计、历史案例、代码差异）
        ↓
📝 manual-case-generator
        ↓
📄 结构化手工测试用例
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

第四阶段：自动化案例生成与执行 ✅
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📄 手工测试用例 + 📦 规范化产出物
        ↓
🧪 api-case-generator
   辅助工具：
   - Udoc2Code MCP（生成接口代码）
   - Proxy MCP（提供抓包逻辑）
        ↓
🎯 自动化测试用例集
        ↓
📈 执行结果分析
```

## 🚀 预期使用方式（4月20日后）

### 完整自动化执行

```bash
# 一键执行完整模式（所有四阶段）
/qa-full ./project-root

# 指定文档目录
/qa-full \
  --requirement-dir ./docs/requirements \
  --design-dir ./docs/designs \
  --code-dir ./src

# 等价于手动执行以下流程：
# [第一阶段] 规范化所有原始文档
# [第二阶段] /requirement-validator
# [第三阶段] 生成手工测试用例
# [第四阶段] /api-case-generator
```

### 分阶段执行（推荐）

```bash
# 阶段1: 规范化
/qa-full --stage normalize

# 人工审查规范化结果...

# 阶段2: 需求检查
/qa-full --stage validate

# 人工审查检查报告...

# 阶段3: 手工案例
/qa-full --stage manual-cases

# 人工审核测试用例...

# 阶段4: 自动化生成
/qa-full --stage automation
```

## 📊 完整模式 vs 快速模式

| 维度 | 快速模式 (`/qa-quick`) | 完整模式 (`/qa-full`) |
|------|------------------------|----------------------|
| **输入要求** | 仅需 KM 开发方案 | 需求 + 设计 + 代码 + 历史案例 |
| **处理时间** | 短（2步，10-20分钟） | 长（4阶段，1-2小时） |
| **测试覆盖** | 接口自动化测试 | 手工 + 自动化 + 需求验证 |
| **质量保证** | 接口级验证 | 全流程验证 |
| **产出物** | API 自动化用例 | 检查报告 + 手工用例 + 自动化用例 |
| **当前状态** | ✅ 可用 | 🚧 开发中 |
| **推荐场景** | 接口变更、快速迭代 | 新功能开发、重大变更 |

## 💡 当前可用替代方案

在完整模式开发完成前，可以手动执行部分阶段：

### 方案 1: 手动执行第二阶段（需求检查）

```bash
# 准备文档目录
mkdir -p ./requirement_word ./design_word

# 将需求和设计文档放入对应目录
cp 需求文档*.docx ./requirement_word/
cp 设计文档*.docx ./design_word/

# 执行需求验证
/requirement-validator

# 查看生成的检查报告
# ./result/需求实现检查报告.docx
```

### 方案 2: 使用快速模式

如果主要关注接口测试，推荐使用快速模式：

```bash
/qa-quick ./docs/development-plan.md
```

### 方案 3: 组合使用

```bash
# 1. 需求检查
/requirement-validator

# 2. 快速生成接口测试
/qa-quick ./docs/development-plan.md

# 结合两种模式的优势
```
